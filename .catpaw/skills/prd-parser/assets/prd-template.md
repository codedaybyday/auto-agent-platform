## 各章节填写说明

### 一、需求背景

**目标**：让读者快速理解「为什么做」「解决什么问题」「目标用户是谁」。

**要求**：
- 2-4 句话，不超过 200 字
- 必须回答三个核心问题
- 避免复制原文大段内容，提炼要点

**示例**：
> 为提升商家运营效率，解决批量修改商品价格耗时过长的问题，需要开发「批量改价」功能。目标用户为美团外卖商家，预计每日节省运营时间 2 小时。

---

### 二、功能点清单

**目标**：完整列出所有功能点，边界清晰。

**要求**：
- 不遗漏任何功能点，哪怕是「新增一个按钮」「修改文案」这类小改动
- 功能点之间边界清晰，不要把多个功能混在一起描述
- 迭代需求需在功能名称前标注 `[新增]`、`[修改]`、`[删除]`
- 多个子需求时，功能点名称前加子需求标签，如 `[活动创建] 功能点 1`

**示例**：
```
### 功能点 1：[新增] 批量选择商品

- **描述**：商家可在商品列表中多选商品，支持全选和反选
- **涉及角色**：商家
- **关键业务规则**：最多支持 100 个商品同时选中
- **原型图参考**：商品列表页 - 批量选择态
```

---

### 三、验收标准

**目标**：具体可验证，避免模糊表述。

**要求**：
- 每个验收条件必须可验证
- 描述清楚「什么情况下算通过」
- 不要写模糊的「功能正常运行」

**对比**：
| ❌ 错误 | ✅ 正确 |
|--------|--------|
| 功能正常运行 | 用户点击提交按钮后，3 秒内显示成功提示，数据写入数据库 |
| 页面加载快 | 页面首屏加载时间 < 2 秒 |
| 交互流畅 | 列表滚动帧率 ≥ 50fps |

---

### 四、注意事项与风险点

**目标**：识别潜在问题和依赖项。

**要求**：
- 原文有明确说明的直接提取
- 原文没有但可以合理推断的，在条目末尾注明「（推断）」
- 关注边界条件、异常处理、兼容性、外部依赖

**示例**：
```
- 批量操作时需考虑大商品量的性能问题，建议前端分页加载（推断）
- 依赖商品中心接口，需确认接口稳定性
- 需兼容 Chrome 80+ 和 Safari 13+
```

---

### 五、待确认问题

**目标**：找出真正模糊、矛盾或缺失的信息。

**要求**：
- 这是报告最有价值的部分之一，认真对待
- 不要为了填充而捏造问题
- 也不要因为文档看起来完整就跳过
- 如果确实没有问题，写「暂无待确认问题」

**好问题的特征**：
- 原文描述前后矛盾
- 关键业务规则缺失
- 交互细节不明确
- 数据结构定义模糊

**示例**：
```
- [ ] 批量选择上限是 100 个商品，还是 1000 个？原文「批量选择」章节写 100，「注意事项」写 1000，存在矛盾
- [ ] 商品排序规则是什么？原文未提及
- [ ] 删除商品后，已选中的状态如何处理？原文未提及
```

---

### 六、UI 布局

**目标**：让没看过设计稿的开发同学也能理解页面结构。

**要求**：
- 每张原型图单独一个条目
- 按四个维度描述：整体布局、核心元素、交互状态、关键文案
- 如果一张图对应多个状态（如默认态和选中态），分别描述
- 原型图名称需与功能点中的「原型图参考」对应

**描述维度详解**：

| 维度 | 说明 | 示例 |
|------|------|------|
| 整体布局 | 页面整体结构 | 顶部导航 + 内容区 + 底部操作栏 |
| 核心元素 | 主要 UI 组件及功能 | 搜索框：支持按商品名称搜索 |
| 交互状态 | 图中展示的状态 | 默认态：商品未选中；选中态：商品已勾选 |
| 关键文案 | 按钮、提示、标签文字 | 「全选」「取消选择」「确认批量改价」 |

**示例**：
```
### 商品列表页 - 批量选择态

**整体布局**：顶部筛选栏 + 中部商品列表 + 底部操作栏

**核心元素**：
- 筛选栏：包含搜索框、分类筛选、状态筛选
- 商品列表：卡片式布局，每张卡片含复选框、商品图片、名称、价格
- 底部操作栏：固定底部，含「全选」「取消选择」「确认批量改价」按钮

**交互状态**：图中展示选中态，3 个商品已勾选，复选框为蓝色填充状态

**关键文案**：
- 按钮文案：「全选」「取消选择」「确认批量改价(3)」
- 提示文案：「已选择 3 个商品」
```
