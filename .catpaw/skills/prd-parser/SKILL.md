# 需求文档解析 Skill

从学城 KM 文档中读取 PRD/需求文档，解析原型图 UI 布局，按标准模板输出结构化的需求分析报告。

> **核心设计**：解析步骤与本文件分离，输出模板在 `assets/prd-template.md`。解析前请先读取该文件了解输出结构。
>
> **用户扩展**：解析前扫描 `references/custom-*.md`，将其中的额外解析规则、输出章节追加到解析流程中执行。

---

## 前置准备

确保 `km` CLI 工具可用：

```bash
source ~/.meituan-local-tools/.venv/bin/activate
```

如果 `km` 命令不存在，参考 km-doc-tools skill 安装说明。

---

## 解析流程

### 第零步：确认需求文档是否已提供

执行解析前先判断用户是否提供了可用的需求文档输入（满足任一即可）：
- 学城 KM 文档链接
- 本地需求文档路径或附件
- 直接粘贴的需求正文

**如果未提供**：
- 不要开始解析，不要臆测需求细节
- 先向用户确认并请其补充需求文档
- 可使用以下话术：
  - `我可以继续做需求解析，但当前还没有收到需求文档。请提供 KM 链接、本地文档路径/附件，或直接粘贴需求正文。`

仅在拿到需求文档后，再继续后续步骤。

### 第一步：读取输出模板

```bash
# 读取模板文件，了解最终输出的章节结构
```

读取 `assets/prd-template.md`，了解六个章节的结构和填写要求。后续所有解析内容都需按此模板组织。

---

### 第二步：获取文档内容

从用户提供的 URL 中提取 doc_id：

| URL 格式 | doc_id 提取 |
|----------|-------------|
| `km.sankuai.com/collabpage/2747985261` | `2747985261` |
| `km.sankuai.com/page/12345678` | `12345678` |

```bash
source ~/.meituan-local-tools/.venv/bin/activate
km get <doc_id>
```

**处理长文档**：
- 文档 ≤200 行：直接解析
- 文档 >200 行：先通读全文理清结构，再逐章节解析
- 有嵌入子文档链接时，按需用 `km get` 递归读取关键子文档

---

### 第三步：获取业务知识与自定义知识

在解析需求前，需要先获取相关的业务知识和自定义知识，以便更好地理解需求背景、业务术语、领域概念以及项目特定的规范和约定。

**并行调用两个 subagent**：

#### Subagent 1：获取业务知识

调用 `general-agent`，从本地知识库 `.catpaw/knowledge_hub/` 获取业务相关的知识：

```json
{
  "subagent_type": "general-agent",
  "description": "获取业务知识",
  "prompt": "你是业务知识获取专家。请执行以下任务：\n\n1. 首先检查 .catpaw/knowledge_hub/ 目录是否存在，如果不存在则返回空结果\n\n2. 读取 .catpaw/knowledge_hub/INDEX.md 了解知识库结构和检索决策树\n\n3. 根据需求主题，从知识库中查找相关的业务知识文档：\n   - L2-modules.md: 功能模块知识，查找相关业务模块\n   - L1-architecture.md: 工程架构，了解技术架构背景\n   - ORG-1.md / ORG-2.md: 组织级基建库，了解团队和公司级规范\n   - L0-spec.md: 项目规范和约定\n\n4. 整理并输出以下信息：\n   - 业务术语和概念解释\n   - 相关业务流程\n   - 系统架构概览\n   - 需求中可能涉及的业务概念清单\n\n注意：如果知识库不存在或无相关文档，返回空结果并说明原因。"
}
```

#### Subagent 2：获取自定义知识

调用 `general-agent`，专门负责从项目的 `references/custom-*.md` 文件获取自定义扩展知识：

```json
{
  "subagent_type": "general-agent",
  "description": "获取自定义知识",
  "prompt": "你是项目自定义知识获取专家。请执行以下任务：\n\n1. 扫描项目 .catpaw/skills/prd-parser/references/ 目录下的所有 custom-*.md 文件\n\n2. 读取所有找到的自定义知识文件内容\n\n3. 整理并输出以下信息：\n   - 自定义解析规则（如果有）\n   - 额外的输出章节要求（如果有）\n   - 项目特定的规范和约定（如果有）\n   - 其他扩展知识（如果有）\n\n注意：如果没有找到任何 custom-*.md 文件，返回空结果并说明原因。"
}
```

**合并知识并应用**：

等待两个 subagent 完成后，合并它们的输出：

1. **业务知识**（来自 Subagent 1，来源：`.catpaw/knowledge_hub/`）：
   - 在解析文档时参考业务术语解释
   - 理解业务流程和系统架构
   - 识别需求中涉及的业务概念

2. **自定义知识**（来自 Subagent 2）：
   - 将自定义解析规则追加到后续解析流程中
   - 将额外的输出章节要求合并到模板结构中
   - 遵循项目特定的规范和约定

**注意事项**：
- 两个 subagent 应并行调用以提高效率
- 如果任一知识获取失败或无相关内容，继续执行后续解析步骤，但需在输出中标注
- 业务知识优先从本地 `.catpaw/knowledge_hub/` 目录读取，无需远程调用

---

### 第四步：解析文档内容

阅读文档，识别以下核心结构（不同文档结构不同，灵活判断）：

**必提取项**：
- **背景/目标**：为什么做，解决什么问题
- **用户故事/场景**：谁在什么情况下需要什么
- **功能描述**：具体做什么，包括交互逻辑、业务规则
- **数据/接口**：涉及的数据结构、API 变更
- **非功能需求**：性能、安全、兼容性等
- **验收标准**：怎么判断需求完成了

**迭代需求标注**：
- 功能点名称前标注 `[新增]`、`[修改]`、`[删除]`
- 多个子需求时，功能点名称前加子需求标签

---

### 第五步：解析原型图

文档中的原型图是理解需求的关键，必须单独处理。

**1. 识别图片链接**

`km get` 返回的 Markdown 中，图片格式如下：
```
![图片描述](https://km.sankuai.com/api/file/cdn/...)
```

注意表格「示意图」列中可能嵌套图片链接。

**2. 读取图片内容**

```bash
source ~/.meituan-local-tools/.venv/bin/activate
km read-file <图片URL>
```

`km read-file` 返回图片的 base64 编码，可用于视觉分析。

**3. 使用视觉能力识别 UI 布局**

对每张原型图调用视觉能力，按以下四个维度描述：

| 维度 | 说明 | 示例 |
|------|------|------|
| 整体布局 | 页面整体结构 | 顶部导航 + 内容区 + 底部操作栏 |
| 核心元素 | 主要 UI 组件 | 筛选栏、商品卡片列表、悬浮按钮 |
| 交互状态 | 图中展示的状态 | 默认态/选中态/空态/加载态 |
| 关键文案 | 重要文字标注 | 按钮文字、提示信息、标签 |

**4. 关联功能点**

在对应功能点描述中补充「参见原型图：[页面/模块名称]」，将 UI 描述与功能点关联。

**异常处理**：
- 图片无法读取（链接失效或权限不足）：注明「原型图链接无法访问」
- 图片不是原型图（如流程图、架构图）：按实际内容描述，不强套 UI 布局模板

---

### 第六步：生成需求英文名

需求英文名用于创建输出目录，命名规则：

1. **去掉前缀标签**：移除 `【PRD】`、`【PC/App】`、`【B端】` 等方括号内容
2. **翻译核心关键词**：将剩余中文标题翻译成英文，提取核心关键词，不必逐字翻译
3. **格式化**：kebab-case，全部小写，**20 个字符以内**

**示例**：
| 原标题 | 英文名 |
|--------|--------|
| `【PRD】免单活动页-新增会员购组件` | `free-order-member-buy` |
| `【PC/App】虚假折扣菜治理起送价一键调整工具` | `fake-dish-min-order` |
| `【B端】商家营销活动权限管控` | `merchant-promo-perm` |

---

### 第七步：输出解析结果

**输出路径**：`{项目根目录}/.catpaw/docs/{需求英文名}/prd.md`

```bash
# 1. 获取当前项目根目录
pwd

# 2. 创建输出目录
mkdir -p {项目根目录}/.catpaw/docs/{需求英文名}

# 3. 写入解析报告
# 将按模板组织的内容写入 prd.md
```

**写入完成后**：
- 告知用户完整文件路径
- 简要概述需求要点（背景 + 核心功能点数量 + 原型图数量）
- 自动启动 subagent 调用 `prd-reviewer` 进行质量评估

### 第八步：自动执行质量评估

**自动触发**：需求解析完成后，立即使用 subagent 调用 `prd-reviewer` skill 进行质量评估，无需用户确认。

**评估参数**：
- 原始 PRD 文档：用户提供的 KM 链接或内容
- 结构化文档路径：`{项目根目录}/.catpaw/docs/{需求英文名}/prd.md`

**调用方式**：使用 `task` 工具启动 `general-agent` subagent，执行 prd-reviewer 评估流程：

```json
{
  "subagent_type": "general-agent",
  "description": "PRD质量评估",
  "prompt": "你是 PRD 质量评估专家。请执行 prd-reviewer skill 对刚生成的结构化需求文档进行质量评估。

任务要求：
1. 读取原始 PRD 文档内容（用户提供的 KM 链接或已缓存的原始内容）
2. 读取刚生成的结构化文档：{项目根目录}/.catpaw/docs/{需求英文名}/prd.md
3. 按三维度评估：内容还原度、PRD 质量、结构化质量
4. 输出评估报告到：{项目根目录}/.catpaw/docs/{需求英文名}/prd-review.md

请严格按照 prd-reviewer skill 的流程执行评估，输出完整的评估报告。"
}
```

**评估流程**：
1. 读取原始 PRD 内容（从 KM 链接或缓存）
2. 读取刚生成的结构化文档
3. 按三维度评估（内容还原度、PRD 质量、结构化质量）
4. 输出评估报告到同目录 `prd-review.md`

**输出示例**：
```
✅ 需求解析完成！

📄 结构化文档: .catpaw/docs/fake-dish-min-order/prd.md
📊 评估报告: .catpaw/docs/fake-dish-min-order/prd-review.md

📈 总分: 85/100 (基本通过)

发现 2 个建议改进项:
- 验收标准"功能正常运行"过于模糊,建议具体化
- 缺少接口依赖说明

详细评估结果请查看评估报告。
```

#### 8.1 执行质量评估

使用 `task` 工具启动 `general-agent` subagent，执行 prd-reviewer 评估流程：

**评估流程**：
1. 并行调用3个subagent执行评估：
   - Subagent 1: 原始PRD分析（提取核心信息、识别PRD问题）
   - Subagent 2: 内置标准评估（内容还原度、PRD质量、结构化质量）
   - Subagent 3: 业务自定义标准评估
2. 汇总评估结果并计算总分
3. 输出评估报告到同目录 `prd-review.md`

#### 8.2 执行自动修复

**修复触发条件**：评估总分 < 90 分时，自动执行修复流程。

**修复策略**：根据评估报告中的问题列表，按优先级自动修复结构化文档：

| 问题类型 | 修复方式 |
|---------|--------|
| ❌ 缺失信息 | 从原始 PRD 中提取缺失内容并补充 |
| ⚠️ 错误信息 | 按原始 PRD 内容修正错误描述 |
| ℹ️ 冗余信息 | 删除不必要的添加内容 |
| 原始 PRD 缺少信息 | 自动标注「待补充」并记录原因 |
| 业务规则不明确 | 仅在报告中标注，需人工处理 |

**自动修复执行步骤**：

1. **读取评估报告**：解析 `prd-review.md` 中的问题列表
2. **问题分类**：按严重程度排序（🔴 必须修复 > 🟡 建议修复 > 🔵 可选修复）
3. **依次修复**：
   - 对于缺失信息：重新解析原始 PRD，提取相关内容
   - 对于错误信息：对照原始 PRD 修正描述
   - 对于冗余信息：删除不必要的添加内容
4. **更新文档**：修改 `prd.md` 文件
5. **记录修复日志**：在 `prd-review.md` 中添加修复记录

**自动修复 subagent 调用**：

```json
{
  "subagent_type": "general-agent",
  "description": "PRD自动修复",
  "prompt": "你是 PRD 文档修复专家。请根据评估报告自动修复结构化文档。\n\n输入信息：\n1. 原始 PRD 内容：{原始PRD内容}\n2. 结构化文档路径：{项目根目录}/.catpaw/docs/{需求英文名}/prd.md\n3. 评估报告路径：{项目根目录}/.catpaw/docs/{需求英文名}/prd-review.md\n\n修复流程：\n1. 读取评估报告，提取所有问题\n2. 按优先级排序问题（必须修复 > 建议修复 > 可选修复）\n3. 对于每个可自动修复的问题：\n   - 读取原始 PRD 相关内容\n   - 修正结构化文档对应部分\n4. 更新结构化文档 prd.md\n5. 在评估报告中添加修复记录章节\n\n修复原则：\n- 只修复有明确原始 PRD 依据的问题\n- 对于原始 PRD 未提及的信息，标注「待补充」\n- 对于业务规则不明确的问题，保留原样，需人工处理\n\n请执行修复并输出修复摘要。"
}
```

#### 8.3 循环验证（可选）

**触发条件**：首次修复后总分仍 < 85 分时，执行二次验证修复。

**验证流程**：
1. 重新调用 prd-reviewer 进行评估
2. 如果发现新问题，继续修复
3. 最多循环 2 次，避免无限循环
