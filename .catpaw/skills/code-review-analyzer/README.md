# Code Review Analyzer - Skill 使用说明

## 功能概述

这个 skill 帮助开发者系统地评审代码，确保业务逻辑与 PRD 需求对齐，技术实现符合技术文档规范。

**核心特性：**
- 🎯 聚焦评审当前分支与 master 的代码差异
- 📚 支持美团学城文档自动获取
- 🔧 支持自定义技术栈（nine、高达等）
- 📊 生成结构化评审报告，与代码同级保存
- ✅ 提取 PRD 需求列表，便于对照检查遗漏功能

## 使用方法

### ⚠️ 必需信息

使用此 skill 时，**必须提供以下信息**：

1. **PRD文档链接**：学城链接或页面ID
2. **技术方案链接**：学城链接或页面ID

技术栈、分支信息等会自动检测，无需手动提供。

### 基本用法

**完整链接格式：**
```
请帮我评审代码，PRD链接：https://km.sankuai.com/page/123456，技术方案链接：https://km.sankuai.com/page/789012
```

**简化格式（直接提供页面ID）：**
```
评审代码，PRD：123456，技术方案：789012
```

### 带业务说明（可选）

可以添加业务背景说明，帮助更好地理解评审目标：

```
评审支付功能的代码，这是一个对接微信支付的模块。PRD：123456，技术方案：789012
```

### 指定技术栈（可选）

技术栈会自动从 package.json 检测，但也可以手动指定：

```
评审代码，PRD：123456，技术方案：789012，技术栈：nine
```

### 错误示例

❌ **缺少必需信息：**
```
帮我评审代码
```
→ 将提示必须提供 PRD 和技术方案链接

❌ **只提供一个链接：**
```
评审代码，PRD：123456
```
→ 将提示缺少技术方案链接

## 输出文档

### 1. PRD 需求分析文档 (`PRD_REQUIREMENTS_[日期].md`)

这份文档会先从 PRD 中提取需求列表，并提供技术设计建议：

```markdown
# PRD 需求分析

**文档来源：** https://km.sankuai.com/page/123456
**分析日期：** 2024-03-19

## 一、需求列表

### 1.1 功能需求

| 需求ID | 需求名称 | 需求描述 | 优先级 | 状态 |
|--------|---------|---------|--------|------|
| REQ-001 | 用户登录 | 支持手机号+验证码登录 | P0 | 待实现 |
| REQ-002 | 密码登录 | 支持账号密码登录 | P1 | 待实现 |
| REQ-003 | 第三方登录 | 支持微信、支付宝登录 | P2 | 待实现 |

### 1.2 非功能需求

| 需求ID | 需求类型 | 需求描述 | 标准 |
|--------|---------|---------|------|
| NFR-001 | 性能 | 登录响应时间 | < 1秒 |
| NFR-002 | 安全 | 密码加密存储 | bcrypt加密 |

## 二、业务流程分析

### 2.1 主流程
1. 用户进入登录页面
2. 选择登录方式（手机号/密码/第三方）
3. 输入凭证信息
4. 系统验证
5. 登录成功，跳转首页

### 2.2 分支流程
- 验证码发送失败 → 提示重新发送
- 密码错误 → 提示重试，3次后锁定
- 第三方授权失败 → 提示稍后重试

### 2.3 异常流程
- 网络异常 → 显示友好提示
- 服务不可用 → 显示维护页面

## 三、业务规则

### 3.1 数据校验规则
| 规则ID | 校验项 | 规则描述 | 错误提示 |
|--------|--------|---------|---------|
| RULE-001 | 手机号 | 11位数字，1开头 | 请输入正确的手机号 |
| RULE-002 | 验证码 | 6位数字 | 验证码格式错误 |

### 3.2 计算规则
| 规则ID | 计算项 | 计算逻辑 | 示例 |
|--------|--------|---------|------|
| RULE-001 | 登录超时 | 当前时间 - 最后登录时间 > 30天 | 强制重新登录 |

### 3.3 权限规则
| 规则ID | 角色 | 权限范围 | 限制条件 |
|--------|------|---------|---------|
| RULE-001 | 普通用户 | 基础功能 | 无 |
| RULE-002 | VIP用户 | 全部功能 | 需验证会员身份 |

## 四、边界情况

| 场景ID | 场景描述 | 预期处理 | 触发条件 |
|--------|---------|---------|---------|
| EDGE-001 | 账号被锁定 | 提示联系客服 | 连续错误3次 |
| EDGE-002 | 验证码过期 | 提示重新获取 | 超过5分钟 |
| EDGE-003 | 设备切换 | 需要二次验证 | 检测到新设备 |

## 五、技术设计建议

### 5.1 架构设计
**建议方案：**
采用前后端分离架构，前端负责UI交互，后端提供RESTful API

**关键模块：**
- 认证模块：负责用户身份验证
- 会话模块：负责登录状态管理
- 第三方对接模块：负责第三方登录集成

### 5.2 数据模型设计
**主要实体：**
```
User {
  id: string - 用户唯一标识
  phone: string - 手机号
  password: string - 加密密码
  loginType: enum - 登录方式
  lastLoginTime: timestamp - 最后登录时间
}
```

### 5.3 接口设计
**接口列表：**

| 接口名称 | 请求方法 | 路径 | 说明 |
|---------|---------|------|------|
| 发送验证码 | POST | /api/auth/sms | 发送登录验证码 |
| 手机登录 | POST | /api/auth/login/sms | 手机验证码登录 |
| 密码登录 | POST | /api/auth/login/password | 账号密码登录 |

### 5.4 关键技术点
1. **验证码防刷**
   - 实现建议：使用Redis限流，同一手机号60秒内只能发送一次
   - 注意事项：需要处理Redis失效情况

2. **密码安全**
   - 实现建议：使用bcrypt加密，cost设置为10
   - 注意事项：禁止明文存储，禁止在日志中打印

### 5.5 性能优化建议
1. 使用Redis缓存用户会话信息
2. 登录验证接口添加限流保护
3. 第三方登录使用异步回调处理

### 5.6 安全性建议
1. 所有接口使用HTTPS
2. 敏感接口添加签名验证
3. 登录失败记录日志用于安全审计

## 六、测试要点

### 6.1 功能测试
- [ ] 手机号验证码登录成功
- [ ] 账号密码登录成功

### 6.2 边界测试
- [ ] 验证码过期处理
- [ ] 连续错误锁定账号
- [ ] 设备切换二次验证

### 6.3 性能测试
- [ ] 1000并发登录响应时间
- [ ] 验证码发送成功率
```

## 评审流程

1. **获取文档**：从学城获取PRD和技术文档
2. **提取需求**：生成需求分析文档，列出所有需求和边界情况
3. **获取代码差异**：获取当前分支与master的diff
4. **技术栈检测**：自动检测项目使用的技术栈
5. **对照评审**：将代码实现与需求列表逐一对照
6. **生成报告**：输出评审报告，包含问题列表和改进建议

## 支持的技术栈

- Nine：美团内部框架
- Gundam（高达）：美团组件化框架
- React：前端框架
- Vue：前端框架

## 注意事项

1. 确保 PRD 和技术文档的学城链接有效
2. 评审前建议先提交代码到当前分支
3. 生成的需求分析文档可用于后续需求变更对比
4. 评审报告会标注必须修改、建议修改、可选优化三个级别
