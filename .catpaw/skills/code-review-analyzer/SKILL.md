# Code Review Analyzer

一个专业的代码评审分析工具，帮助开发者系统地评审代码，确保业务逻辑与PRD需求对齐，技术实现符合技术文档规范。

**核心特性：**
- 🎯 聚焦评审当前分支与 master 的代码差异
- 📚 支持美团学城文档自动获取
- 🔧 支持自定义技术栈（nine、高达等）
- 📊 生成结构化评审报告，与代码同级保存

## 使用场景

当用户有以下需求时，主动使用此 skill：
- 评审当前分支相对于 master 的代码变更
- 检查代码是否符合PRD需求
- 验证技术实现是否符合技术文档
- 进行代码审查和需求对齐
- 分析业务流程的正确性
- 对比 PRD 与代码实现的一致性

## 工作流程

### 第一步:收集必要信息

**⚠️ 必需信息（必须提供）：**

以下信息是评审的必要条件，**如果用户未提供，必须主动询问并暂停后续流程**：

1. **PRD文档链接** - 学城文档链接或页面ID
   - 格式：`https://km.sankuai.com/page/xxx` 或页面ID数字
   - 用于：提取业务需求、业务流程、边界情况等

2. **技术方案链接** - 学城文档链接或页面ID
   - 格式：`https://km.sankuai.com/page/xxx` 或页面ID数字
   - 用于：对照技术架构设计、API契约、数据模型等

**自动化检测信息（可选）：**

以下信息会自动检测，用户无需提供：

- **技术栈类型**：从 `package.json` 自动识别（检测 `nine`、`gundam`、`react`、`vue` 等关键词）
- **当前分支**：自动获取 Git 当前分支名
- **代码差异**：自动计算当前分支与 master 的 diff
- **项目根目录**：默认为当前工作目录

**如果用户未提供必需信息，立即停止并询问：**

```
⚠️ 缺少必要信息，无法开始评审。

请提供以下必需信息：
1. **PRD文档链接**（必填）：学城链接或页面ID
   示例：https://km.sankuai.com/page/123456 或直接提供页面ID 123456

2. **技术方案链接**（必填）：学城链接或页面ID
   示例：https://km.sankuai.com/page/789012 或直接提供页面ID 789012

提供以上信息后，我将立即开始代码评审。
```

**示例输入：**

完整格式：
```
评审代码，PRD链接：https://km.sankuai.com/page/123456，技术方案链接：https://km.sankuai.com/page/789012
```

简化格式：
```
评审代码，PRD：123456，技术方案：789012
```

带技术栈说明（可选）：
```
评审支付功能的代码，PRD：123456，技术方案：789012，技术栈：nine
```

**并行执行以下任务：**

#### 2.1 获取学城文档（纯文档分析，不涉及代码）

```bash

km_search "搜索关键词"

1. **提取 PRD 核心信息（仅基于PRD文档）：**
   - 核心业务需求
   - 业务流程步骤
   - 业务规则和条件判断
   - 用户角色和权限要求
   - 边界场景和异常处理需求
   - 功能符合度要求

对比 PRD 和技术方案，识别差异：

1. **PRD有但技术方案没有的内容** → ⚠️ 需要补充技术设计
2. **技术方案有但PRD没有的内容** → 忽略（可能是过往版本信息）
3. **两者都有但描述不一致** → ❌ 标记为差异点，需要确认

#### 2.2 获取代码差异

执行 Git 命令获取当前分支与 master 的差异：

```bash
# 获取当前分支名
git branch --show-current

# 获取与 master 的文件差异列表
git diff master --name-status

# 获取详细的代码差异
git diff master

# 如果需要，获取新增文件的完整内容
git diff master --diff-filter=A --name-only | xargs -I {} sh -c 'echo "=== {} ===" && cat {}'

# 获取 master 分支的代码内容（用于检查需求是否在历史代码中）
git show master:path/to/file
```

#### 2.3 检测技术栈

从 `package.json` 自动检测技术栈：

```bash
# 读取 package.json
cat package.json

# 根据依赖判断技术栈：
# - 包含 "nine" 相关依赖 → nine 技术栈
# - 包含 "gundam" 相关依赖 → 高达技术栈
# - 包含 "react" → React 技术栈
# - 包含 "vue" → Vue 技术栈
```

### 第三步:评审变更代码

**前置工作：确认文档已生成并验证历史代码**
- 检查是否已存在以下文档：
  - `PRD_REQUIREMENTS_[日期].md`：需求分析文档（纯文档分析结果）
  - `TECH_SPEC_[日期].md`：技术方案分析文档
  - `TECH_DESIGN_[日期].md`：技术设计建议文档
- 如果不存在，先生成这些文档
- **对比 PRD 和技术方案**，识别差异点

**⚠️ 重要：验证需求文档中的缺失功能是否在历史代码中已存在**

在文档生成完成后，执行历史代码验证：

1. **识别文档中的缺失功能**：
   - 从 `PRD_REQUIREMENTS_[日期].md` 中提取所有需求
   - 标记为"待实现"的需求

2. **检查历史代码**：
   ```bash
   # 在 master 分支中搜索相关关键词
   git grep "功能关键词" master
   
   # 查看相关文件的历史版本
   git show master:path/to/file
   
   # 搜索函数名、类名等
   git grep -E "function|class|const" master -- '*.js' '*.ts'
   ```

3. **更新需求文档**：
   - 如果发现需求在历史代码中已实现：
     - 更新需求状态为「📦 历史代码已实现」
     - 添加代码位置：`file:line (master)`
     - 添加说明：「该功能在历史代码中已实现，本次无需开发」
   - 示例更新：
     ```markdown
     | REQ-001 | 用户登录 | 📦 历史代码已实现 | [`login.js:45 (master)`](login.js:45) | 已在历史版本实现 |
     ```

4. **重新统计需求状态**：
   - 更新需求状态统计表
   - 计算真实的缺失需求数量
   - 更新功能覆盖率

**代码评审阶段才开始结合代码**

现在开始结合代码进行评审，重点关注 diff 中新增和修改的部分，对照更新后的需求分析文档进行验证：

#### 1. PRD业务逻辑对照

**核心业务需求验证:**
- 代码是否实现了PRD定义的所有业务场景
- 用户角色和权限控制是否正确实现
- 业务条件判断是否完整(如库存检查、资格验证等)

**需求缺失检查流程：**

当发现某个需求在当前 diff 中未实现时，执行以下检查：

1. **检查历史代码**：
   ```bash
   # 在 master 分支中搜索相关关键词
   git grep "关键词" master
   
   # 查看相关文件的历史版本
   git show master:path/to/file
   ```

2. **判断实现状态**：
   - ✅ **已实现**：在当前 diff 中找到了实现
   - 📦 **历史代码中存在**：在 master 分支的历史代码中已实现，当前 diff 未涉及
   - ❌ **未实现**：在当前分支和历史代码中都未找到实现

3. **标注说明**：
   - 如果需求在历史代码中已实现，标注为「📦 历史代码中存在」
   - 提供历史代码的位置：`file:line (master)`
   - 说明该需求并非缺失，而是已在历史代码中实现

**功能符合度检查:**
- 列出PRD要求的功能点,逐一验证是否实现
- 检查业务计算逻辑(价格、折扣、库存等)是否准确
- 验证状态流转是否符合PRD定义

**业务流程逻辑:**
- 绘制代码实现的业务流程
- 与PRD描述的流程对比,找出差异
- 识别未覆盖的边界场景

**条件判断完整性:**
- 检查所有if/else分支是否符合PRD规则
- 验证异常情况的提示和处理
- 确认边界条件的处理逻辑

#### 2. 业务规则与技术实现对照

**业务规则实现:**
- PRD中的每条业务规则是否都有对应的代码实现
- 折扣、优惠、规则计算是否准确
- 规则优先级和组合逻辑是否正确

**功能一致性:**
- 是否遗漏PRD中定义的必备功能
- 功能实现是否与PRD描述存在偏差
- 用户交互流程是否符合预期

**错误处理与边界情况:**
- 列出PRD定义的边界情况,检查是否都有处理
- 错误提示是否友好、准确
- 异常流程是否能正常恢复或退出

#### 3. 技术文档一致性

**架构与设计一致性:**
- 代码结构是否符合技术文档的架构设计
- 是否采用了推荐的技术方案
- 模块划分和依赖关系是否合理

**API契约:**
- 接口路径、方法、参数是否符合文档定义
- 请求和响应数据格式是否规范
- 错误码和状态码是否一致

#### 4. 代码质量

**可读性与可维护性:**
- 业务逻辑代码是否清晰易懂
- 是否存在过度复杂的逻辑需要重构
- 命名是否准确反映业务含义
- 是否有充分的注释说明关键业务逻辑

**测试覆盖:**
- 核心业务逻辑是否有单元测试
- 边界情况是否有测试覆盖
- 测试用例是否覆盖PRD定义的主要场景

**可扩展性:**
- 代码是否易于添加新的业务规则
- 配置和规则是否可灵活调整
- 是否考虑了未来可能的需求变化

#### 5. 性能

**性能瓶颈:**
- 识别可能存在性能问题的代码段
- 检查数据库查询是否合理
- 是否有N+1查询、循环调用等问题

**数据处理优化:**
- 复杂计算是否可以优化
- 是否有不必要的重复计算
- 数据批量处理是否合理

#### 6. 安全性

**安全检查:**
- 用户输入是否经过验证和过滤
- 敏感数据是否正确加密存储
- 是否存在SQL注入、XSS等安全漏洞

**权限控制:**
- 是否正确验证用户权限
- 用户是否只能访问自己的数据
- 关键操作是否有权限检查

### 第四步:生成评审报告

**生成的文档：**

1. **需求分析文档**：`PRD_REQUIREMENTS_[日期].md`
   - 简洁的需求清单（ID、名称、描述、PRD位置）
   - 业务流程概览
   - 业务规则和边界情况
   - 链接到技术设计建议文档

2. **技术方案分析文档**：`TECH_SPEC_[日期].md`
   - 架构设计概览
   - 接口列表
   - 数据模型
   - 链接到技术方案原文

3. **技术设计建议文档**：`TECH_DESIGN_[日期].md`
   - 架构建议
   - 关键技术点及示例代码
   - 性能和安全性建议
   - 测试要点

4. **代码评审报告**：`CODE_REVIEW_REPORT_[分支名]_[日期].md`
   - 变更文件清单
   - PRD需求对照（简表）
   - PRD与技术方案差异分析
   - 主要问题列表（含源代码和修改建议）
   - 改进建议汇总

所有文档均保存在项目根目录，相互关联，避免内容重复。

使用以下结构生成评审报告（详细模板见 `assets/report_template.md`）：

**报告结构（精简版）：**

1. **分支信息和文档来源**（单行展示）
2. **变更文件清单**（列表形式，可点击）
3. **需求实现对照**（统计表+关键需求表）
4. **PRD与技术方案差异**（仅列出差异项）
5. **主要问题**（分三类：必须修改、建议修改、可选优化）
   - 每个问题包含：位置、源代码、修改建议
6. **总体评估**（评分+对齐度+改进建议）

**报告特点：**
- ✅ 聚焦问题发现和改进建议
- ✅ 所有代码位置可点击跳转
- ✅ 详细需求和技术设计在独立文档
- ✅ 避免内容重复，保持简洁

**示例片段：**

```markdown
# 代码评审报告

**分支信息：**
- 当前分支: [分支名]
- 对比基准: master
- 评审日期: [YYYY-MM-DD]

### 1.3 删除文件

| [文件1] | [删除原因] |

## 二、项目背景

### 2.1 项目概述

> **参考文档：** `PRD_REQUIREMENTS_[日期].md`

### 3.1 核心业务需求对照

> **参考文档：** `PRD_REQUIREMENTS_[日期].md`
> **技术方案：** `TECH_SPEC_[日期].md`

**需求实现状态统计：**

| 状态 | 数量 | 占比 |

## 评审技巧

### 需求分析准确性检查

**防止虚假需求：**
- 在提取需求时，确保每个需求都有明确的PRD原文支撑
- 对于相似的需求，合并而不是拆分
- 如果PRD中某段内容已被其他需求覆盖，不要重复提取
- 需求ID应该是连续的，不应该有跳号

**需求验证步骤：**
1. 提取需求时，标注PRD原文位置
2. 检查是否有重复或相似的需求
3. 确保每个需求都是独立且必要的
4. 验证需求之间的依赖关系是否合理

### 关注业务逻辑核心
- 优先评审PRD中定义的核心业务流程
- 重点检查条件判断和边界处理
- 验证业务规则的计算逻辑
- **对照技术方案验证实现是否符合设计**

### 三文档对照验证方法
- **PRD → 需求列表**：确保需求提取准确，无虚假需求
- **技术方案 → 实现设计**：验证代码实现符合技术设计
- **PRD vs 技术方案**：识别差异，重点关注不一致项
- 绘制流程图对比业务流程
- 列出边界情况清单，逐一检查

### 代码阅读技巧
- 从入口函数开始,跟踪业务流程
- 关注条件分支和异常处理
- 检查数据流转和状态变化

### 输出清晰明确
- 使用具体的代码位置引用(file:line)
- 给出明确的问题描述和建议
- 用数据说话,给出覆盖率等量化指标

## 注意事项

1. **保持客观** - 基于文档和代码事实,避免主观判断
2. **关注重点** - 优先评审核心业务逻辑和安全相关代码
3. **提供建设性意见** - 不仅指出问题,还要提供解决方案
4. **量化评估** - 尽可能用数据量化评审结果
5. **关注实际价值** - 关注真正影响用户体验和系统稳定性的问题
