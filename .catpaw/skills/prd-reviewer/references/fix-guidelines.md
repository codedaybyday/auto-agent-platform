# PRD 自动修复指南

本文档说明如何根据评估报告自动修复结构化文档中发现的问题。

---

## 一、修复原则

### 1.1 以原始 PRD 为准

**修复内容必须来自原始 PRD 文档，不能凭空添加。**

- ✅ 正确：从原始 PRD 中提取缺失的内容并补充
- ❌ 错误：根据理解自行编造内容

### 1.2 保守修复原则

**无法确定的内容，宁可标注「待确认」也不要猜测。**

- 原始 PRD 中没有的信息 → 标注「待补充」
- 无法确定修复方式的 → 标注「待确认」
- 涉及核心业务规则的修改 → 需用户确认

### 1.3 最小修改原则

**只修复发现的问题，不主动优化其他内容。**

- ✅ 正确：只修复评估报告中的问题
- ❌ 错误：顺便优化其他描述

### 1.4 可追溯原则

**所有修复都要有标注说明原因和时间。**

```markdown
<!-- 自动修复: 补充原始 PRD 中的登录方式描述 - 2024-01-15 -->
```

---

## 二、问题分类与处理方式

### 2.1 可自动修复的问题

直接修改，无需确认：

| 问题类型 | 修复方式 | 示例 |
|---------|--------|------|
| ❌ 缺失信息 | 从原始 PRD 中提取缺失内容并补充 | 补充功能点遗漏的验收标准 |
| ⚠️ 错误信息 | 按原始 PRD 内容修正错误描述 | 修正业务规则描述错误 |
| ℹ️ 冗余信息 | 删除不必要的添加内容 | 删除原文未提及的功能描述 |
| 功能点粒度过粗 | 拆分为合理粒度的功能点 | 将包含多个独立功能的点拆开 |
| 功能点粒度过细 | 合并过于细碎的功能点 | 将同一流程的功能点合并 |
| UI 描述不清晰 | 补充更详细的布局描述 | 添加组件结构、状态说明 |
| UI 状态遗漏 | 补充缺失的状态描述 | 添加空态、加载态、异常态 |

### 2.2 需用户确认后修复的问题

先询问用户，再执行修复：

| 问题类型 | 处理方式 |
|---------|--------|
| 原始 PRD 本身缺少信息 | 询问用户是否需要标注「待补充」，或提供补充内容 |
| 验收标准模糊 | 展示原始表述，询问用户如何具体化 |
| 逻辑矛盾 | 展示矛盾点，询问用户选择哪个规则 |
| 功能点边界模糊 | 展示重叠点，询问用户如何划分 |

### 2.3 无法自动修复的问题

仅在报告中标注，不修改文档：

| 问题类型 | 原因 |
|---------|------|
| 原始 PRD 信息不完整 | 无法从原文提取，需人工补充 |
| 业务规则不明确 | 需与产品确认 |
| 缺少验收标准 | 需与产品确认具体标准 |
| 外部依赖不明确 | 需与相关方确认 |

## 三、修复执行流程

### 3.1 准备阶段

1. 读取原始 PRD 文档内容
2. 读取结构化文档 `prd.md` 内容
3. 解析评估报告中的问题列表

### 3.2 修复阶段

```
for each 问题 in 问题列表:
    if 问题属于「可自动修复」:
        a. 定位问题所在章节/段落
        b. 从原始 PRD 提取正确信息
        c. 生成修复后的内容
        d. 添加修复标注
        e. 替换原内容
    else if 问题属于「需确认后修复」:
        展示问题详情，询问用户
        根据用户选择执行修复或跳过
    else:
        在报告中标注为「需人工处理」
```

### 3.3 完成阶段

1. 更新结构化文档 `prd.md`
2. 更新评估报告 `prd-review.md`，标记已修复的问题
3. 输出修复摘要

## 四、修复标注规范

### 4.1 标注格式

```markdown
<!-- 自动修复: [修复原因] - [修复时间] -->
```

### 4.2 标注位置

**新增内容**：在新增内容前一行添加标注

```markdown
<!-- 自动修复: 补充原始 PRD 中的登录方式描述 - 2024-01-15 -->
支持以下登录方式：
- 账号密码登录
- 手机验证码登录
- 第三方登录
```

**修改内容**：在修改处添加标注

```markdown
<!-- 自动修复: 修正约束条件描述 - 2024-01-15 -->
约束条件：同时满足以下 5 个条件（原描述缺少"同时"）
```

**删除内容**：保留删除标注

```markdown
<!-- 自动修复: 删除原文未提及的功能 - 2024-01-15 -->
<!-- 已删除：自动推荐相关商品功能 -->
```

## 五、修复示例

### 5.1 补充缺失信息

**问题**：功能点缺少登录方式描述

**原始 PRD 内容**：
> 用户可通过账号密码、手机验证码、微信、支付宝等方式登录

**修复前**：
```markdown
### 1. 用户登录

支持用户登录功能。
```

**修复后**：
```markdown
### 1. 用户登录

<!-- 自动修复: 补充原始 PRD 中的登录方式描述 - 2024-01-15 -->
支持用户登录功能，包括：
- 账号密码登录
- 手机验证码登录
- 第三方登录（微信、支付宝）
```

---

### 5.2 修正错误信息

**问题**：业务规则描述错误

**原始 PRD 内容**：
> 约束起送价计算：最终起送价 = Max{目标起送价, 约束起送价}

**修复前**：
```markdown
### 起送价计算

最终起送价 = 目标起送价 和 约束起送价 的平均值
```

**修复后**：
```markdown
### 起送价计算

<!-- 自动修复: 修正起送价计算公式 - 2024-01-15 -->
最终起送价 = Max{目标起送价, 约束起送价}
```

---

### 5.3 删除冗余信息

**问题**：添加了原文未提及的功能

**原始 PRD 内容**：无"自动推荐"相关描述

**修复前**：
```markdown
### 5. 自动推荐相关商品

系统根据用户购买记录自动推荐相关商品。
```

**修复后**：
```markdown
<!-- 自动修复: 删除原文未提及的功能 - 2024-01-15 -->
<!-- 已删除：功能点 5「自动推荐相关商品」，原文未提及 -->
```

---

### 5.4 拆分功能点

**问题**：功能点粒度过粗，包含多个独立功能

**修复前**：
```markdown
### 1. 起送价推荐策略

圈选受影响商家，计算推荐起送价。
```

**修复后**：
```markdown
<!-- 自动修复: 拆分功能点粒度 - 2024-01-15 -->

### 1. 受影响商家圈选

基于历史订单数据圈选受影响商家（需同时满足 5 个条件）。

### 2. 起送价计算

在约束条件下计算推荐起送价（四步流程）。
```

```markdown
- ❌ 缺失信息: Y 个
- ⚠️ 错误信息: Z 个
- ℹ️ 冗余信息: W 个

📝 修复内容摘要:
1. [功能点/用户登录] 补充登录方式描述
2. [功能点/订单管理] 修正状态流转逻辑
3. [待确认问题] 删除无效问题（原文已有答案）

2. [异常处理] 原文未说明接口调用失败如何处理，请补充

📄 评估报告已更新: .catpaw/docs/fake-dish-min-order/prd-review.md
```

## 七、特殊情况处理

### 7.1 原始 PRD 信息不完整

**处理方式**：
1. 在结构化文档中标注 `<!-- 待补充: 原始 PRD 未提及 XXX -->`
2. 在评估报告中标注为「需人工处理」
3. 给出建议补充的内容方向

**示例**：
```markdown
<!-- 待补充: 原始 PRD 未说明接口调用失败如何处理，建议补充失败提示文案和重试机制 -->
```

### 7.2 修复后引入新问题

**处理方式**：
1. 回滚本次修复
2. 在评估报告中标注修复失败原因
3. 将问题标记为「需人工处理」

### 7.3 多处同一问题

**处理方式**：
1. 识别所有同类问题位置
2. 统一修复
3. 在修复摘要中说明修复位置数量
