# 技术方案评审标准

本文档定义了技术方案评审的详细标准，供 `tech-design-reviewer` skill 使用。

---

## 一、需求覆盖度评审标准（满分 10 分）

### 1.1 功能点覆盖（6 分）

**满分条件**：PRD 中每一个功能点都有对应的技术实现描述。

| 扣分项 | 扣分 |
|--------|------|
| 每遗漏一个主要功能点 | -2 分 |
| 每遗漏一个次要功能点（如文案修改、样式调整） | -0.5 分 |
| 每遗漏一个边界条件或异常流程的处理 | -1 分 |
| 验收标准中有条目未在技术方案中体现 | -0.5 分/条 |

**判断方法**：
- 将 PRD 功能点清单与技术方案逐一对照
- 「功能点」包括：新增功能、修改功能、删除功能、UI 变更、业务规则变更
- 「边界条件」包括：数量上限、状态约束、权限控制、并发处理

### 1.2 需求理解准确性（3 分）

**满分条件**：技术方案对需求的理解与 PRD 完全一致，无误解或偏差。

| 扣分项 | 扣分 |
|--------|------|
| 业务规则理解错误（如数量限制写错） | -2 分/处 |
| 状态枚举或字段定义与 PRD 不符 | -1 分/处 |
| 功能逻辑理解偏差（实现了错误的功能） | -2 分/处 |
| 用户角色或权限理解错误 | -1 分/处 |

### 1.3 过度设计识别（1 分）

**满分条件**：技术方案不包含 PRD 未要求的功能扩展或不必要的复杂度。

| 扣分项 | 扣分 |
|--------|------|
| 引入了 PRD 未提及的功能扩展（未注明「扩展点」） | -0.5 分/处 |
| 存在明显的过早优化（如 PRD 无性能要求却设计了复杂缓存） | -0.5 分/处 |

## 二、项目现实对齐评审标准（满分 10 分）

> 此维度需要读取项目知识库或代码才能评审。如无法获取项目上下文，跳过此维度并在报告中注明。

### 2.1 技术栈符合性（3 分）

**满分条件**：方案中使用的所有技术选型与项目实际一致。

| 扣分项 | 扣分 |
|--------|------|
| 框架/库版本与项目实际不一致 | -1.5 分/处 |
| 引入了项目中未使用的新依赖（未说明引入原因） | -1 分/个 |
| 技术选型与项目规范冲突（如项目禁止使用的库） | -2 分/处 |

**判断方法**：
- 对比方案中的技术选型与项目 `package.json`/`pom.xml`/`go.mod` 中的实际依赖
- 检查项目知识库中是否有技术选型禁止项或必选项

### 2.2 目录结构和模块划分符合性（3 分）

**满分条件**：方案中新增的文件路径、模块职责划分与项目现有规范完全一致。

| 扣分项 | 扣分 |
|--------|------|
| 新增文件路径不符合项目目录规范 | -1 分/处 |
| 模块职责划分与项目现有架构不一致（如业务逻辑放错层） | -2 分/处 |
| 有可复用的现有模块但方案选择了重新实现 | -1 分/处 |

**判断方法**：
- 对比方案中的文件路径与项目实际目录结构
- 搜索项目中是否已有类似功能的实现

### 2.3 接口和数据库规范符合性（2 分）

**满分条件**：方案中的接口路径风格、字段命名、数据库约定与项目现有规范完全一致。

| 扣分项 | 扣分 |
|--------|------|
| 接口路径风格与项目现有接口不一致 | -0.5 分/处 |
| 字段命名规范与项目不一致 | -0.5 分/处 |
| 数据库设计不符合项目惯例（如缺少必要字段） | -1 分/处 |

**判断方法**：
- 对比方案接口路径与项目已有接口的路径风格
- 对比方案字段命名与项目已有表的字段命名规范

### 2.4 与现有实现的一致性（2 分）

**满分条件**：方案与现有代码无逻辑冲突，影响范围评估准确。

| 扣分项 | 扣分 |
|--------|------|
| 方案与现有代码存在逻辑冲突 | -1.5 分/处 |
| 方案修改的接口/字段有其他调用方但影响范围未覆盖 | -1 分/处 |

**判断方法**：
- 搜索项目中对方案修改的接口/字段的引用
- 检查方案新增字段是否与现有字段语义重叠

## 三、技术栈规范符合性评审标准（满分 10 分）

> 此维度需要从 knowledge_hub 获取技术栈知识库才能评审。如无法获取相关技术栈知识库，跳过此维度并在报告中注明。

### 3.1 框架/库使用规范符合性（4 分）

**满分条件**：方案中对框架和库的使用完全符合知识库中的规范和最佳实践。

| 扣分项 | 扣分 |
|--------|------|
| API 调用方式错误（如 React hooks 使用不规范） | -2 分/处 |
| 生命周期函数使用不当 | -1.5 分/处 |
| 状态管理方案不符合团队规范 | -1.5 分/处 |
| 使用了已废弃的 API 或方法 | -1 分/处 |

**判断方法**：
- 对照知识库文档中的 API 使用规范
- 检查知识库中是否有明确的使用禁忌
- 验证方案中的技术实现是否符合知识库推荐的最佳实践

### 3.2 组件使用规范符合性（3 分）

**满分条件**：方案中对组件的使用完全符合组件库文档规范。

| 扣分项 | 扣分 |
|--------|------|
| 组件 API 误用（props、events、slots 使用错误） | -1.5 分/处 |
| 使用了已废弃的组件或组件属性 | -1 分/处 |
| 存在更优组件但方案选择了次优方案 | -0.5 分/处 |
| 不必要的组件封装或过度封装 | -0.5 分/处 |

**判断方法**：
- 对照组件库知识库文档中的 API 说明
- 检查知识库中是否有组件选择建议
- 验证方案中的组件组合方式是否符合推荐模式

### 3.3 最佳实践符合性（3 分）

**满分条件**：方案完全遵循知识库中的最佳实践建议。

| 扣分项 | 扣分 |
|--------|------|
| 安全相关实现不符合安全规范 | -2 分/处 |
| 性能优化方案不符合最佳实践 | -1 分/处 |
| 错误处理方式不符合团队规范 | -1 分/处 |
| 代码组织方式与团队约定不一致 | -0.5 分/处 |

**判断方法**：
- 对照知识库中的最佳实践章节
- 检查知识库中是否有明确的安全/性能规范
- 验证方案是否遵循了知识库中的推荐模式

---

## 四、方案可落地性评审标准（满分 10 分）

### 4.1 接口设计具体性（3 分）

**满分条件**：每个接口都有明确的路径、方法、请求/响应结构。

| 扣分项 | 扣分 |
|--------|------|
| 接口路径不明确（只写「新增查询接口」） | -1 分/个 |
| 请求参数缺少字段定义或类型说明 | -0.5 分/个接口 |
| 响应结构缺少字段定义 | -0.5 分/个接口 |
| 缺少错误码说明 | -0.5 分 |

### 4.2 代码改动具体性（3 分）

**满分条件**：涉及的文件路径、函数名、模块职责都有明确说明。

| 扣分项 | 扣分 |
|--------|------|
| 只写「修改前端组件」，未指明具体文件 | -1 分/处 |
| 只写「新增 Service 方法」，未说明方法签名 | -0.5 分/处 |
| 前端/后端改动层次不完整（缺少某一层的说明） | -1 分/层 |
| 数据库变更无 DDL 语句或字段定义 | -1 分 |

### 4.3 工作量估算合理性（2 分）

**满分条件**：每个功能点有工作量估算，且估算合理。

| 扣分项 | 扣分 |
|--------|------|
| 完全没有工作量估算 | -2 分 |
| 部分功能点缺少工作量估算 | -0.5 分/处 |
| 工作量估算明显不合理（如复杂功能估算 0.5 天） | -1 分/处 |

### 4.4 不确定项标注（2 分）

**满分条件**：所有需要进一步确认的技术决策都已标注「待确认」。

| 扣分项 | 扣分 |
|--------|------|
| 存在明显的技术选型争议点但未标注 | -1 分/处 |
| 依赖外部团队确认的事项未标注 | -1 分/处 |

## 五、工程质量评审标准（满分 10 分）

### 5.1 接口设计规范（2.5 分）

**满分条件**：接口设计符合项目规范（或通用 RESTful 规范），字段命名规范，鉴权方式明确。

> 注意：如果项目有自己的接口规范，以项目规范为准，不要用通用 RESTful 规范去否定项目的既有约定。

**检查项**：
- HTTP 方法使用是否正确（GET 查询、POST 创建、PUT/PATCH 更新、DELETE 删除）
- 路径命名是否符合项目规范（或通用 kebab-case）
- 请求/响应字段命名是否符合项目规范（或通用 camelCase）
- 是否说明了鉴权方式（Token、Cookie、无需鉴权等）
- 分页接口是否有 pageNum/pageSize 等标准参数
- 列表接口是否有排序参数说明

| 扣分项 | 扣分 |
|--------|------|
| HTTP 方法使用错误（如用 GET 做写操作） | -1 分/处 |
| 路径命名不规范 | -0.5 分/处 |
| 缺少鉴权说明 | -0.5 分 |
| 分页/排序参数不规范 | -0.5 分 |

### 5.2 数据库设计规范（2.5 分）

**满分条件**：表结构设计合理，考虑了兼容性和性能，有数据迁移方案（如需）。

**检查项**：
- 新增字段是否有默认值（避免 NOT NULL 无默认值导致存量数据问题）
- 索引设计是否合理（高频查询字段是否有索引，是否有冗余索引）
- 字段类型是否合适（如用 VARCHAR 存枚举值是否合理）
- 是否考虑了数据量增长对性能的影响
- 有存量数据时是否有数据迁移方案
- 字段命名是否符合项目规范（或通用 snake_case）

| 扣分项 | 扣分 |
|--------|------|
| 新增 NOT NULL 字段无默认值 | -1 分/处 |
| 高频查询字段无索引 | -1 分/处 |
| 有存量数据但无迁移方案 | -1 分 |
| 字段命名不规范 | -0.5 分/处 |

### 5.3 影响范围分析完整性（3 分）

**满分条件**：从代码、接口、数据库、依赖、兼容性五个维度完整分析影响范围。

**检查项**：

**代码影响**（0.6 分）：
- 是否列出了所有需要修改的文件/模块
- 是否说明了每个文件的改动类型（新增/修改/删除）

**接口影响**（0.6 分）：
- 是否列出了所有新增/修改/废弃的接口
- 修改的接口是否说明了向前兼容性

**数据库影响**（0.6 分）：
- 是否列出了所有表结构变更
- 是否说明了变更对现有数据的影响

**依赖影响**（0.6 分）：
- 是否说明了对上游服务的影响（调用方需要做什么）
- 是否说明了对下游服务的影响（被调用方需要做什么）
- 是否说明了第三方依赖的变更

**兼容性影响**（0.6 分）：
- 是否考虑了客户端版本兼容（App 多版本并存）
- 是否考虑了接口向前兼容（老版本客户端调用新接口）
- 是否考虑了数据格式兼容

| 扣分项 | 扣分 |
|--------|------|
| 完全缺少某个维度的分析 | -0.6 分/维度 |
| 某维度分析不完整（有遗漏） | -0.3 分/维度 |

### 5.4 风险评估充分性（2 分）

**满分条件**：识别了主要技术风险，每个风险有应对措施，高风险改动有回滚方案。

**必须识别的风险类型**（根据改动类型判断是否适用）：

| 风险类型 | 适用场景 |
|---------|---------|
| 性能风险 | 涉及大数据量查询、高并发写入 |
| 数据一致性风险 | 涉及多表事务、分布式操作 |
| 兼容性风险 | 修改已有接口、数据库字段变更 |
| 安全风险 | 涉及权限控制、用户数据、支付 |
| 依赖风险 | 依赖外部服务、第三方 API |
| 回滚风险 | 有数据迁移、不可逆操作 |

| 扣分项 | 扣分 |
|--------|------|
| 遗漏了明显适用的风险类型 | -0.5 分/类型 |
| 风险有识别但无应对措施 | -0.5 分/处 |
| 高风险改动无回滚方案 | -1 分 |
| 无灰度/分批上线策略（对于高风险改动） | -0.5 分 |

## 六、综合评分规则

五个维度各满分 10 分，综合评分计算方式：

```
综合评分 = (需求覆盖度 × 0.25) + (项目现实对齐 × 0.20) + (技术栈规范符合性 × 0.20) + (方案可落地性 × 0.25) + (工程质量 × 0.10)
```

**权重说明**：
- 需求覆盖度（25%）：技术方案首要职责是覆盖需求
- 项目现实对齐（20%）：方案必须符合项目实际，否则开发无法按方案执行
- 技术栈规范符合性（20%）：方案必须符合技术栈规范和最佳实践，避免引入技术债务
- 方案可落地性（25%）：方案必须具体到可指导开发的程度
- 工程质量（10%）：工程规范重要但可以在开发中逐步修正

**评分等级**：

| 综合评分 | 等级 | 是否可用于开发 |
|---------|------|--------------|
| 9-10 分 | 优秀 | ✅ 可直接开发 |
| 7-8.9 分 | 良好 | ✅ 可直接开发（关注建议项） |
| 5-6.9 分 | 一般 | ⚠️ 修复必须项后可开发 |
| 3-4.9 分 | 较差 | ⚠️ 需要较多修复后才能开发 |
| 0-2.9 分 | 不合格 | ❌ 需重新设计 |

## 七、问题严重程度分级

### 🔴 必须修复（影响开发正确性）

以下情况必须在开发前修复：
- 功能点遗漏（开发会漏做功能）
- 需求理解错误（开发会做错功能）
- 技术栈与项目实际不一致（开发无法按方案执行）
- 模块职责划分与项目架构冲突（会引入架构负债）
- 框架/库使用规范违反（可能导致 bug 或性能问题）
- 组件 API 误用（可能导致功能异常）
- 安全相关实现不符合规范（存在安全风险）
- 接口设计缺失（前后端无法联调）
- 数据库变更无兼容性考虑（可能导致线上故障）
- 高风险操作无回滚方案

### 🟡 建议修复（影响开发效率）

以下情况建议在开发前修复，否则会增加沟通成本：
- 接口/数据库命名规范与项目不一致（需要额外对齐）
- 有可复用模块但方案选择重新实现（增加工作量）
- 接口字段说明不完整（开发需要反复确认）
- 工作量估算缺失（排期困难）
- 影响范围分析不完整（可能遗漏改动点）
- 不确定项未标注（可能在开发中产生争议）
- 非最佳实践的技术实现（可能影响性能或可维护性）
- 组件选择次优（可能影响开发效率或用户体验）

### 🟢 可选优化

以下情况可以在后续迭代中改进：
- 过度设计（不影响当前需求交付）
- 工程规范小问题（不影响功能正确性）
- 文档格式和可读性问题
- 性能优化建议（当前性能已满足需求）

## 八、特殊场景处理

### 纯前端需求

重点关注：
- 组件拆分方案是否合理，是否符合项目组件规范
- 状态管理设计是否清晰，是否复用了项目现有状态管理模式
- 接口联调方案是否完整
- 性能优化方案（如有大列表、频繁渲染）
- 组件使用是否符合组件库规范

可以降低对数据库设计的要求（如无数据库改动）。

### 纯后端/接口需求

重点关注：
- 接口设计是否完整规范，是否符合项目接口风格
- 数据库设计是否合理，是否符合项目数据库约定
- 性能和并发处理方案

可以降低对前端改动的要求。

### 重构/优化需求

重点关注：
- 改动范围边界是否清晰（哪些不动）
- 回归测试范围是否明确
- 灰度/分批上线策略

需求覆盖度评审重点变为「重构目标是否达成」而非「功能点覆盖」。

### 无项目上下文的情况

如果用户未提供项目路径，且知识库和代码都无法获取：
- 跳过「项目现实对齐评审」维度
- 在报告中注明「因缺少项目上下文，无法进行项目现实对齐评审」
- 综合评分调整权重：需求覆盖度 30%，技术栈规范符合性 25%，方案可落地性 30%，工程质量 15%

### 无 PRD 的情况

如果用户只提供了技术方案，没有 PRD：
- 跳过「需求覆盖度评审」维度
- 在报告中注明「因缺少 PRD，无法进行需求覆盖度评审」
- 综合评分调整权重：项目现实对齐 30%，技术栈规范符合性 25%，方案可落地性 30%，工程质量 15%

### 无技术栈知识库的情况

如果 knowledge_hub 中没有找到相关技术栈的知识库文档：
- 跳过「技术栈规范符合性评审」维度
- 在报告中注明「因缺少技术栈知识库，无法进行技术栈规范符合性评审」
- 综合评分调整权重：需求覆盖度 30%，项目现实对齐 25%，方案可落地性 30%，工程质量 15%

### 同时缺少多个评审维度的情况

如果同时缺少多个维度的评审依据，按以下规则处理：
- 最多跳过 2 个维度，如果缺少 3 个及以上维度，告知用户无法进行有效评审
- 综合评分权重按剩余维度等比例调整
- 必须在报告中清晰说明跳过的维度和原因
