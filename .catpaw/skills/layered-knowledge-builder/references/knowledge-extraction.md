# 第二步：提取知识条目（详细步骤）

从"已确认知识"范围内提取知识条目，严格遵守以下规则：

- 用户明确提供的内容可以直接提取，并标注来源为"用户输入"
- 系统自动推导且被用户采纳的内容可以提取，但必须保留原始来源引用，并标注来源为"补充知识（已确认）"
- 系统自动推导但未确认的内容，禁止提取
- 每个条目必须标注：所属层级、知识标题、原文片段或来源摘录、来源类型
- 没有对应依据片段或来源摘录的条目不得提取
- 识别条目间的依赖关系，但不因依赖关系补充不存在的条目
- 最佳实践约束、任务契约、Context 文件说明、验收标准、AI 护栏都属于可提取的知识条目类型，不得因为"不是业务功能"而忽略

提取结果格式：
```
层级: L0
标题: 技术栈清单
来源类型: 用户输入 / 补充知识（已确认）/ 代码推导
依据片段: 「用户输入中的原始文字，或已确认补充知识中的直接摘录，或代码推导的文件路径与结构依据」
```

对于"补充知识（已确认）"来源，还必须额外保留以下元信息，便于后续追溯：

- 候选确认轮次
- 主来源 / 辅来源
- 与项目的关联锚点
- 用户确认结果

## 2.1 上下文工程类条目的额外提取要求

若来源材料中出现以下内容，必须主动判断是否可形成独立知识条目，而不是并入普通说明文字：

- 明确的任务契约模板或字段清单
- 明确规定 AI 先读哪些文件、后做哪些动作的执行顺序
- 明确的 Stop Checklist、Hook、Guardrail、回滚条件、禁止修改清单
- 明确要求维护的上下文文件及其字段结构
- 明确要求的自验证动作与失败处理机制

输出这类条目时，标题应体现"约束"或"验收"属性，例如：

- L0：AI 任务契约字段要求
- L1：项目级 Context 文件清单与职责分工
- L3：开发完成前自检流程
- L4：质量门禁与失败回退标准
- L6：Harness 最佳实践与常见偏航纠偏

## 2.2 知识条目四要素标注（可选增强）

对于 L2（功能模块）、L6（工程经验）以及上下文工程类条目，建议在提取时补充四要素标注，使知识条目更易被 AI 后续精准检索和使用：

```
实体（Entity）: 该知识描述的核心对象，如组件名、接口名、模块名、函数名、规则名
关系（Relation）: 该实体与其他实体的关联，如「依赖」「调用」「继承」「约束」「替代」
约束（Constraint）: 使用该实体时的规范或限制，如「不得修改」「必须配合 X 使用」「仅在 Y 场景适用」
上下文（Context）: 该知识生效的场景或边界条件，如「仅限移动端」「仅在登录态下」「迭代 N 之后」
```

四要素标注规则：

- 四要素均为可选，但 L2 条目至少填写实体和关系，L6 条目至少填写约束和上下文
- 实体字段不得填写泛化描述（如"组件"），必须是具体名称
- 约束字段若为空，说明该知识无使用限制，可明确写「无」
- 上下文字段若为空，说明该知识全局适用，可明确写「通用」
- 代码推导来源的条目，四要素置信度默认降一级，需在字段后标注「（推断）」
