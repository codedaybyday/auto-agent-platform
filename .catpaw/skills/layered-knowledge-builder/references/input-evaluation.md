# 第一步：评估输入完整性（详细步骤）

收到用户输入后，按以下两段式流程评估信息完整性。先判断用户显式提供的信息是否足以继续，再自动推导并补齐与当前项目高度关联的候选知识，提交给用户确认是否采纳。

## 1.0 项目类型自动识别

在做完整性评估之前，先根据用户输入（或仓库结构）自动识别项目类型，后续的层级重点提示和知识提取策略将依此调整。

识别依据优先级：用户明确说明 > 仓库依赖清单（package.json / build.gradle / Cargo.toml 等）> 目录结构特征 > 技术栈关键词

项目类型与对应层级重点：

| 项目类型 | 识别特征 | 层级重点调整 |
|----------|----------|-------------|
| 前端 Web | React/Vue/Angular、webpack/vite、src/components | L2 重点：组件说明、页面路由、接口调用约定 |
| 后端服务 | Spring/Express/Gin、controller/service/dao 目录 | L2 重点：接口契约、数据模型、服务依赖关系 |
| 全栈 | 同时含前后端目录或 monorepo 结构 | L1 重点：前后端边界划分；L2 前后端分别建条目 |
| 移动端 | React Native/Flutter/iOS/Android 特征 | L2 重点：页面导航、原生能力调用约定、平台差异 |
| 基础库/SDK | 无业务页面、主要导出 API、有 publish 配置 | L0 重点：API 设计约定；L6 重点：breaking change 记录 |
| 未能识别 | 特征不明显或信息不足 | 按通用策略执行，在输出中标注「类型待确认」 |

识别结果必须在第一步输出中明确告知用户，格式如下：

```
🔍 项目类型识别：[前端 Web / 后端服务 / 全栈 / 移动端 / 基础库 / 未能识别]
识别依据：[说明依据来源]
层级重点：[说明本次生成将重点关注哪些层级]
```

如果识别结果与用户预期不符，用户可直接纠正，后续按用户指定类型执行。

## 1.1 基础输入完整性评估

先基于用户显式输入做完整性检查，输出格式如下：

```
当前信息评估结果：

✅ 已提供：[列出已有信息]
❓ 建议补充（按优先级）：
  1. [P1-必填] 项目名称、业务定位、技术栈、仓库地址
  2. [P1-必填] 所属团队、团队TL、项目负责人
  3. [P2-重要] 当前迭代PRD链接、关键业务流程说明
  4. [P2-重要] 编码规范、分支策略、发布流程
  5. [P3-可选] 依赖的公司级组件/框架版本、团队级公共服务
  6. [P3-可选] 已有踩坑记录、性能基线数值
```

P1 信息缺失时，不能直接进入生成阶段，但也不能只停留在"等用户补充"。此时必须继续执行 1.2，尽量从高相关信息源中自动整理候选补充知识，再请用户一起确认。

## 1.2 自动推导高相关补充知识并请求确认

当用户输入不完整，或虽然完整但明显还存在上下文缺口时，必须主动从与当前项目高度相关的信息源中检索、提炼、去重、归类，并形成"候选补充知识清单"，交给用户确认是否采纳到知识库范围内。

这里的目标不是替用户拍板，而是把系统能自动发现的高价值上下文先整理出来，降低用户补充成本。

允许使用的信息源包括但不限于：

- 学城 / km / wiki / collabpage / contentId / pageId 对应文档
- 用户提供的仓库、目录、README、配置文件、接口文档、设计说明
- 项目内已存在的 AGENTS.md、CLAUDE.md、AI_CONTEXT.md、规范文档、历史知识库文件
- 与项目同名、模块同名、团队同名、技术栈同名的内部知识源
- 其他当前任务可合法访问且与项目直接相关的信息源
- 【兜底】当文档缺失或 P1 信息不足时，可从仓库代码结构中推导：目录树、package.json / build.gradle / Cargo.toml 等依赖清单、路由配置、tsconfig / eslint 配置、核心入口文件、README

自动推导时必须遵守以下原则：

- 只允许提炼"与当前项目高度相关"的内容，禁止广撒网式堆砌背景知识
- 每条候选知识都必须附带来源、关联理由、建议归属层级
- 明确区分"用户已明确提供"与"系统自动推导候选"
- 自动推导出的内容默认都处于"待确认"状态，未经用户确认不得直接纳入最终知识库
- 若多个来源表述重复，必须先合并为一条候选，再给用户确认
- 若候选内容存在冲突，必须显式标注冲突点，优先让用户决策
- 不能只罗列标题，必须提炼出对建库有价值的知识摘要
- 候选数量必须受控，宁缺毋滥，优先保留最有价值的少量条目
- 必须把"上下文工程约束"和"验收标准"视为可沉淀知识，而不是只提功能说明

### 1.2.1 高相关性的严格判定规则

只有通过"命名命中 + 内容命中 + 建库价值"综合判断的候选，才能进入待确认列表。为了让执行更稳定，使用评分制而不是纯口头判断。

评分前提如下：

- 先完成基础去噪，过滤明显无关、泛化、过旧、无来源或无法建立项目锚点的内容
- 再对剩余候选逐条打分，禁止边看边随意放行
- 每条候选必须产出可追溯分数，不能只给"高相关/低相关"这种模糊结论

#### 评分模型

总分 100 分，由四部分组成：命名命中 25 分，内容命中 35 分，建库价值 30 分，来源可信度 10 分。

1）命名命中（0~25 分）

- 25 分：直接出现项目名、仓库名、核心模块名，且出现不止一次，或同时命中多个关键标识
- 18 分：直接出现团队名、平台名、脚手架名、组件库名、服务名，且与当前项目存在明确绑定
- 10 分：命中用户已提供的核心技术标识，但还缺项目级直接绑定
- 0 分：没有明确命名锚点，只是看起来像同类项目

2）内容命中（0~35 分）

- 35 分：内容直接描述当前项目的工程结构、模块、交付流程、运行方式、发布约束或质量机制
- 25 分：内容直接描述当前项目正在采用的团队级/公司级规范，并能和项目文档互相印证
- 15 分：内容只解释部分概念、平台或流程，对当前项目有帮助但不够直接
- 0 分：内容偏泛化，无法证明其适用于当前项目

3）建库价值（0~30 分）

- 30 分：属于生成知识库时的关键缺口，不补会明显影响分层归类或后续理解
- 22 分：能稳定沉淀为 ORG-1 / ORG-2 / L0 / L1 / L3 / L4 / L6 中的高复用知识
- 12 分：有一定帮助，但更像背景补充而不是核心沉淀
- 0 分：即使相关，也不值得进入知识库

4）来源可信度（0~10 分）

- 10 分：权威一手来源，如项目仓库主文档、项目内规范、直接对应的学城正式文档、官方平台说明
- 6 分：可信二手来源，如团队经验文档、历史知识库、被多个来源交叉印证的材料
- 3 分：只有单一弱来源，但内容暂时可参考
- 0 分：来源不明、版本过旧、无法验证

#### 准入阈值

- 总分 ≥ 75 分：可进入"建议优先确认"列表
- 总分 60~74 分：可进入"可选补充"列表，但必须明确说明不确定性
- 总分 < 60 分：直接丢弃，不展示给用户

除此之外，还必须满足两个硬门槛：

- 命名命中不得为 0 分
- 建库价值不得低于 12 分

只要任一硬门槛不满足，即使总分够高，也不得进入候选列表。

#### 强制拦截规则

以下情况直接判定为不通过，不进入打分后的展示阶段：

- 只是同技术栈教程、行业通用最佳实践，缺乏项目绑定关系
- 只提到同一团队或同一平台，但无法证明与当前项目直接相关
- 内容陈旧，明显早于项目当前阶段，且无新来源交叉验证
- 来源本身存在明显冲突或失真，无法形成可信摘要
- 无法提炼出清晰的知识摘要、关联锚点和建库价值

#### 执行顺序

执行时必须按以下顺序进行，不得跳步：

1. 候选收集
2. 去噪过滤
3. 逐条打分
4. 应用硬门槛
5. 按分数分桶（建议优先确认 / 可选补充 / 丢弃）
6. 对保留候选做去重与冲突归并
7. 最终生成用户确认清单

#### 评分输出格式

每条保留候选都必须保留如下评分记录：

```
命名命中：25
内容命中：35
建库价值：30
来源可信度：10
总分：100
分桶：建议优先确认 / 可选补充 / 丢弃
```

### 1.2.2 候选知识提炼格式

每条候选知识在进入待确认列表前，都必须先被提炼成统一结构：

```
候选标题：
建议层级：
知识摘要：
来源：
关联锚点：
关联理由：
建库价值：
命名命中分：
内容命中分：
建库价值分：
来源可信度分：
总分：
分桶：建议优先确认 / 可选补充
置信度：高 / 中 / 低
状态：待确认
```

其中：

- "知识摘要"必须是对来源内容的压缩提炼，不得只写标题，不得复制整段原文
- "关联锚点"必须指出它和当前项目之间的直接连接点，例如项目名、仓库配置、模块名、README 语句、依赖项、目录名、平台名
- "建库价值"必须写清楚它为什么值得进入分层知识库，而不是泛泛说"有帮助"
- 分数字段必须与上面的评分模型保持一致，不能凭感觉乱写
- 分桶结果必须由分数和硬门槛共同决定，不能手工改桶
- 置信度为低的候选，默认不进入待确认列表，除非当前确实缺关键上下文且无更高质量来源
- 来源为代码推导时，必须在"来源"字段标注为「代码推导：[文件路径或结构依据]」，并在"知识摘要"中说明是从代码结构推断而非文档明确描述，置信度默认为"中"，不得标为"高"

### 1.2.3 去重、合并与冲突处理规则

在把候选知识展示给用户前，必须先进行一轮归并整理，避免把碎片信息原样倾倒给用户。

去重规则：

- 同一知识点来自多个来源时，按"权威性更高、更新更近、与项目绑定更强"的原则合并为一条
- 标题不同但表达的是同一规则、同一流程、同一平台能力时，必须合并
- 仅补充细节、示例、截图而不改变知识主体的内容，不单独成条

合并后保留：

- 主来源：最权威或最贴近项目的一份来源
- 辅来源：用于交叉验证或补充细节的来源
- 合并摘要：统一后的知识摘要

冲突处理规则：

- 如果不同来源对同一知识点的结论不一致，不能自行拍板，必须生成"冲突候选"
- 冲突候选必须写清：冲突点、各来源说法、推荐用户确认的问题
- 存在冲突的候选，默认不纳入"建议直接采纳"，而是单独放入"需你判断"的分组

### 1.2.4 候选数量控制规则

候选补充知识不是越多越好，必须控制数量，优先给用户看最值得确认的内容。

默认规则如下：

- 首轮最多展示 5 条候选补充知识
- 若高价值候选不足 3 条，可以只展示 1~2 条，禁止为了凑数降低标准
- 若检索结果很多，只保留 Top 5，排序优先级为：总分 > 命名命中分 > 建库价值分 > 来源可信度分 > 信息新鲜度
- 若存在"需你判断"的冲突候选，最多额外展示 2 条，并与普通候选分组展示
- 超出上限的候选不展示，但可以在内部保留为后备，等用户确认首轮后再继续补充
- "可选补充"分桶的候选，默认排在"建议优先确认"之后，不得混排到前面

### 1.2.5 用户确认时的展示要求

输出给用户确认时，必须先按层级或主题归类，再按优先级排序，不得把原始检索结果直接堆给用户。

展示时每条至少包含：候选标题、建议层级、知识摘要、来源、关联理由、总分、分桶、是否存在冲突。

如果候选较多，优先展示以下类型：

- 项目依赖的 ORG-1 / ORG-2 基建知识
- 当前项目特有的 L0 / L1 规范与架构知识
- 当前项目高频使用的 L3 / L4 流程与质量知识
- 具有稳定复用价值的 L6 经验知识

输出给用户确认时，必须使用类似如下结构：

```
在你已提供的信息之外，我又自动整理出一批与当前项目高度相关的候选补充知识，请你确认是否采纳：

1. [待确认][建议纳入 ORG-1] MachPro 脚手架规范
   - 来源：学城《xxx》
   - 关联理由：你的项目 README 和构建配置都显示该项目基于 MachPro
   - 可补充内容：脚手架约定、构建命令、标准目录约束

2. [待确认][建议纳入 L3] 团队发布流程
   - 来源：学城《xxx》 + 仓库文档《xxx》
   - 关联理由：用户已提供仓库地址，且文档中明确提到该项目沿用该发布流程
   - 可补充内容：提测、发布、回滚、验收检查点

请你按"采纳 1/2"、"只采纳 2"、"都不采纳"或直接补充修改意见回复我。未经你确认，我不会把这些候选知识正式纳入知识库。
```

如果用户未明确回复是否采纳，则不能把候选补充知识并入正式知识条目；此时只能继续基于"用户明确提供 + 用户已确认采纳"的范围执行。

如果用户只做了部分确认，必须显式记录"已采纳 / 未采纳 / 待补充判断"三种状态，后续只允许把"已采纳"部分纳入正式知识输入。

### 1.2.6 上下文工程与 Harness 约束补充规则

知识库不仅是资料归档，也是在为后续 AI 协作建立可执行上下文。因此，凡是能够稳定约束 AI 行为、降低任务偏航、提升可验证性的内容，都应视为"上下文工程知识候选"。

重点关注以下几类内容：

- 任务契约：目标、范围、输入、输出、禁止事项、依赖、完成定义
- 项目级 Context 文件：AGENTS.md、CLAUDE.md、.cursorrules、architecture.md、conventions.md、error-code.md、AI_CONTEXT.md
- 执行护栏：禁止修改目录、必须先读的文档、必须遵守的接口边界、必须声明的 breaking change 条件
- 测试先行与自验证机制：先写用例、执行 lint/type-check/test、失败回滚、结果复核
- 长会话上下文维护：会话开始热身、上下文压缩、活信息维护、关键决策沉淀

这些内容在分层归档时优先按如下思路判断：

- 公司级通用 Harness 规范、统一 AI 协作约束，优先归入 ORG-1
- 团队特有的工程上下文约束、团队级 AI 协作方式，优先归入 ORG-2
- 项目明确采用的任务契约模板、编码约束、禁止事项，优先归入 L0
- 项目上下文文件结构、目录级说明、架构说明文件，优先归入 L1
- 执行流程中的评审、开发、提测、自检、回滚要求，优先归入 L3
- 测试门禁、质量阈值、错误处理、观测要求，优先归入 L4
- 可复用的 Harness 最佳实践、踩坑纠偏、AI 失误防范模式，优先归入 L6

如果某条内容本质上是在约束"AI 应如何理解任务、如何读取上下文、如何证明自己完成任务"，即使它不是业务知识，也应纳入候选补充知识清单。

### 1.2.7 验收标准补充规则

验收标准本身也是知识库的重要组成部分，不能只当作交付后的临时检查项。凡是能够回答"什么叫完成""如何验证完成""哪些情况明确不通过"的内容，都应被提炼成知识条目候选。

提炼验收标准时，优先识别以下层次：

- L1 语法与风格验收：格式、命名、lint、类型检查、目录约定、导出约定
- L2 功能正确性验收：页面行为、交互逻辑、接口调用、边界条件、回归用例
- L3 架构与约束验收：模块边界、依赖方向、禁止改动范围、性能与发布约束、breaking change 兼容性
- L4 业务语义验收：需求目标是否达成、用户场景是否闭环、关键指标与业务规则是否满足

提炼时必须同时写清三类信息：

- 可执行的验证动作：例如运行什么命令、检查什么文件、观察什么行为
- 可量化的通过标准：例如错误数为 0、覆盖率阈值、时延阈值、必须存在的导出项
- 明确的禁止边界：例如不得修改某目录、不得引入 any、不得跳过测试、不得遗漏回滚方案

如果来源中同时包含"自动化验收"和"人工验收"，必须分开提炼，不得混成一句笼统描述。自动化验收优先沉淀到 L3/L4，人工验收更适合作为 L3 流程检查点或 L6 经验补充。

## 1.3 进入后续步骤的准入规则

进入第二步前，知识输入范围必须被划分为两类：

- 已确认知识：用户原始输入中明确提供的内容，以及用户明确同意采纳的候选补充知识
- 待确认知识：系统推导但用户尚未确认、或用户明确拒绝的内容

只有"已确认知识"可以进入后续条目提取与知识库生成流程。

如果 P1 缺失，但已经通过自动推导补齐了一部分高价值候选，也仍需用户确认后才能把这些内容视为有效输入。P1 若在确认后仍缺失，必须暂停，并明确告诉用户还差哪些最小必要信息。
