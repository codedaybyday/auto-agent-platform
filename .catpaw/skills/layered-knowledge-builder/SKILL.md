# 分层知识库架构生成器

面向项目开发场景，基于《前端项目知识库分层架构》规范，从用户提供的项目文档和信息中提取知识，在**本地 Git 项目**中生成完整的分层知识库架构（11 个 Markdown 文件），本地项目是唯一的知识源（Single Source of Truth），可选同步到 BlockAI K-Hub 平台。

> 架构规范参考：[km.sankuai.com/collabpage/2754263079](https://km.sankuai.com/collabpage/2754263079)

---

## 分层架构速查

| 层级 | 名称 | 典型内容 |
|------|------|----------|
| EXTERNAL | 前端社区/官方文档 | 技术栈版本号、选型理由、外链引用 |
| ORG-1 | 公司级基建库 | 脚手架、公共组件库、CI/CD、监控规范、ADR |
| ORG-2 | 团队级基建库 | 团队特有组件、额外SDK、CR规则、偏离说明 |
| L0 | 规范与约定 | 技术栈清单、项目特有编码约定、偏离上层规范说明 |
| L1 | 工程架构 | **技术栈架构（核心框架/状态管理/UI框架/工具库）**、目录结构、模块划分、构建配置、CI/CD差异项 |
| L2 | 功能模块 | 业务模块文档、组件说明、接口约定、PRD/设计稿链接 |
| L3 | 研发流程 | 需求→上线全流程、CR标准、测试策略、发布检查单 |
| L4 | 运维与质量 | 性能基线、监控埋点、错误处理、故障复盘 |
| L5 | 新人入门 | 环境搭建、学习路径图、FAQ（聚合导航层） |
| L6 | 工程经验 | Coding纠偏记录、优化提炼、最佳实践（含反哺机制） |

## 前置检测：block-kb CLI 安装检查

**在执行任何知识库操作前，必须先完成以下检测流程：**

### Step 0：安装并检测 block-kb CLI 版本

#### Step 0-1：检测是否已安装，以及是否为最新版本

```bash
# 获取本地已安装版本（未安装则输出 NOT_INSTALLED）
LOCAL_VERSION=$(block-kb-cli --version 2>/dev/null || echo "NOT_INSTALLED")

# 查询内网 npm 源最新版本
LATEST_VERSION=$(npm view @block/knowledge-cli version \
  --registry http://r.npm.sankuai.com 2>/dev/null || echo "UNKNOWN")

echo "本地版本：$LOCAL_VERSION"
echo "最新版本：$LATEST_VERSION"
```

根据检测结果分三种情况处理：

**情况 A：未安装（NOT_INSTALLED）**

```bash
# 通过 Block AI Market 安装
npx @block/spec-cli@latest block-skill-get 39 --version=0.0.0
# 验证安装结果
block-kb-cli --version
```

**情况 B：已安装但不是最新版本（LOCAL_VERSION ≠ LATEST_VERSION）**

向用户展示提示：

```
⚠️  检测到 @block/knowledge-cli 不是最新版本
   当前版本：<LOCAL_VERSION>
   最新版本：<LATEST_VERSION>

建议更新以获得最新功能（如知识空间管理）。是否立即更新？
  ✅ 是 → 执行 npm install -g @block/knowledge-cli@latest
  ❌ 否 → 继续使用当前版本（部分功能可能不可用）
```

用户确认更新后执行：

```bash
npm install -g @block/knowledge-cli@latest \
  --registry http://r.npm.sankuai.com
block-kb-cli --version  # 确认更新成功
```

> ⚠️ **特别注意**：知识空间（space）功能需要 `@block/knowledge-cli >= 0.1.6`。
> 若用户拒绝更新且当前版本不支持 space 命令，后续空间创建步骤将降级为**仅创建分组**，并告知用户建议更新。

**情况 C：已安装且是最新版本**

```
✅ @block/knowledge-cli 已是最新版本（<VERSION>），无需更新。
```

直接进入 Step 0.5。

> 📦 npm 包名：`@block/knowledge-cli`，命令：`block-kb-cli`
> Block AI Market 页面：[block-kb（ID: 39）](https://block.sankuai.com/ai-market/agents-skills-detail?skills=39)

### Step 0.5：检测 MIS 配置

```bash
block-kb-cli config
```

- 若输出提示「未配置默认 MIS」，执行：`block-kb-cli config --set-mis <用户MIS>`
- 若 token 异常（401），执行：`block-kb-cli config --clear-token` 后重试

### ⚡ L2 功能模块文档专项说明：内置代码扫描能力

**L2 层（功能模块）** 是整个知识库中最贴近业务代码、信息密度最高的层级。本 Skill 内置了完整的代码仓库扫描与业务模块梳理能力，Step 4-6 **优先直接扫描代码仓库生成 L2 文档，无需依赖外部 Skill**。

详细扫描 Prompt 规则见 [references/l2-code-scan-prompt.md](references/l2-code-scan-prompt.md)，输出模板见 [references/l2-module-template.md](references/l2-module-template.md)，示例参考 [examples/l2-sample-guide.md](examples/l2-sample-guide.md)。

#### 触发条件

满足以下任一条件时，直接启动内置代码扫描流程：

- 用户提供了仓库路径或本地代码目录
- 用户提供了仓库地址（可先 clone 到本地再分析）
- 当前会话中已可访问项目代码文件（通过 Explore 代理或工作目录）

若用户未提供仓库/代码，则降级为基于用户文档输入手工提炼 L2 条目，并在输出中注明「L2 文档由文档输入生成，建议后续配合代码仓库重新生成以提升质量」。

#### 内置扫描流程（Step 4-6 执行标准）

**Step 4-6-1：解析项目路径**
- 从用户输入提取仓库路径，默认使用当前工作目录
- 若提供的是远程仓库地址，先 clone 到本地临时目录

**Step 4-6-2：项目类型识别**
- 依据优先级：用户明确说明 > package.json/build.gradle > 目录结构 > 技术栈关键词
- 输出识别结果，告知用户项目类型与层级重点

**Step 4-6-3：代码结构深度扫描**（严格按照 [references/l2-code-scan-prompt.md](references/l2-code-scan-prompt.md) 执行）

1. **目录结构分析**：识别项目类型（React/Vue/Angular等），扫描核心目录，分析配置文件
2. **页面路由提取**：提取所有页面路由和页面组件，按功能模块分组
3. **复杂组件识别与接口提取**：
   - 识别代码量大（>200行）、逻辑复杂、被多处引用的组件，每类不超过5个
   - 对每个复杂组件，必须提取：职责概述、核心能力、Props 列表（名称/类型/是否必填/默认值）、Events/Callbacks、暴露的 Ref 方法
   - 从 TypeScript 类型定义、PropTypes、JSDoc 注释中提取接口信息；如无类型定义则从使用处推断
4. **工具函数/Hooks 分析与接口提取**：
   - 查找复杂的工具函数和自定义 Hooks，分析封装原因和边界情况
   - 对每个复杂函数/Hook，必须提取：函数签名（参数名/类型/返回值）、副作用说明、依赖项（Hook）
5. **全局状态设计与接口提取**：
   - 分析 Redux/Zustand/Context 等状态管理设计，梳理状态模块和同步机制
   - 对每个状态模块，必须提取：对外暴露的选择器或 useXxxStore 方法、触发方式、持久化说明
6. **业务流程推断**：从路由配置、状态管理、API 调用推断核心业务流程，绘制 Mermaid 流程图
7. **特殊标记扫描**：搜索 TODO / FIXME / HACK / NOTE / WARNING，汇总工程技术债务
8. **工程级注意事项**：提取开发部署配置、第三方服务、历史遗留问题等隐性知识

**Step 4-6-4：按 L2 模板生成 `L2-modules.md`**

L2 层唯一输出文件为 **`L2-modules.md`**，内容格式严格按照 [references/l2-module-template.md](references/l2-module-template.md)，汇总**所有业务模块**的工程知识，不得拆散为多个文件。

每个模块必须包含以下三个核心章节（缺一不可）：
- **「功能说明」**：职责概述（整体定位与职责边界）、核心能力（列举关键能力点）、封装原因（为什么需要封装，不直接用原生或第三方库）、核心逻辑（关键处理步骤/算法/方法说明）
- **「接口说明」**：Props/Events（组件）、函数签名含参数与返回值（工具函数/Hook）、对外暴露的选择器或方法及触发方式（状态模块）
- **使用注意事项**：使用限制、边界情况、已知问题（每个模块都必须填写，无注意事项时写「无」）

`L2-modules.md` 完整章节结构：
- **1. 业务流程概览**：核心业务流程图（Mermaid）+ 关键业务规则
- **2. 页面索引**：按功能模块分组的完整页面列表
- **3. 复杂模块说明**：3.1 复杂通用组件 / 3.2 复杂工具函数/Hooks / 3.3 全局状态设计（每类不超过 5 个）
- **4. 工程级注意事项**：开发与部署 / 第三方服务 / 历史问题与技术债务
- **5. 目录结构约定**（如有特殊约定）
- **6. 容器环境适配**（如有多容器环境）

生成原则（来自 [references/l2-code-scan-prompt.md](references/l2-code-scan-prompt.md)）：
- **知识价值性**：只记录重要的、复杂的、容易出错的内容；简单模块无需记录
- **设计意图优先**：说明"为什么"而非仅仅"是什么"，记录技术决策背后的原因
- **注意事项优先**：重点记录使用限制、边界情况、已知问题、容易踩坑的地方
- **标记待确认**：无法确定的信息标记为 `[待确认: xxx]`，推测性内容注明"推测"

**Step 4-6-5：生成后处理（进入 Step 4-7 前必须完成）**

1. **`L2-modules.md` 完整性验证**：确认每个复杂模块都包含「功能说明」「接口说明」「使用注意事项」三个章节；任何模块缺失则必须补全后再继续
2. **层级对齐**：检查内容是否符合 L2 定义（业务模块文档、组件说明、接口约定），过滤越界内容（架构内容 → L1，流程内容 → L3）
3. **四要素补标**：对每个模块条目补充四要素标注（实体 / 关系 / 约束 / 上下文），至少填写实体和关系（见第二步 2.2 节）
4. **待确认项汇总**：所有 `[待确认: xxx]` 项必须在展示给用户前统一汇总，请用户确认后再写入 K-Hub
5. **K-Hub ID 标注**：同其他层级一致，`L2-modules.md` 须通过 K-Hub 检索，匹配到的文档 ID 在文件尾部注释中标注（`<!-- khub: docId=xxx -->`），但**文件正文保留完整内容**

## 知识库渐进迭代机制

知识库不是一次性产物，而是随项目 coding 过程持续演进的活文档。本章节定义在日常 coding 过程中，AI 如何主动发现知识缺口并启发用户补充，使知识库越用越完善。

### 触发场景

以下场景发生时，AI 必须主动提示用户考虑补充知识库，而不是静默推断或跳过：

| 触发场景 | 说明 | 建议补充层级 |
|----------|------|-------------|
| 无锚点推断 | AI 对某个模块、接口、规范的理解只能靠猜测，没有知识库依据 | L0 / L1 / L2 |
| 用户纠偏 | 用户指出 AI 生成的代码或方案有误，且错误根因是知识缺失 | 对应出错的层级 |
| 新技术决策落地 | 项目引入新依赖、新架构模式、新规范，尚未沉淀到知识库 | L0 / L1 / ORG-2 |
| 踩坑复盘 | 出现 bug、性能问题、发布事故，根因分析完成后 | L4 / L6 |
| 重复解释 | 用户在多次会话中反复解释同一背景知识 | 对应知识所属层级 |
| 新成员加入 | 新人提问暴露出知识库中缺少的入门路径或 FAQ | L5 |

### 触发后的引导流程

触发场景发生时，AI 按以下流程处理，不得直接跳过或只在内部记录：

```
1. 明确说明触发原因
   → 「我注意到 [具体情况]，这可能是知识库中缺少 [知识点] 的信号」

2. 提炼候选知识条目
   → 按 1.2.2 格式提炼，标注建议层级、来源、四要素（如适用）
   → 来源类型标注为「迭代补充：[触发场景类型]」

3. 请用户确认是否写入
   → 「是否将以上内容补充到知识库？可以直接说「写入」或告诉我需要调整的地方」

4. 用户确认后执行写入
   → 调用 K-Hub 检索，复用或新建知识库条目
   → 更新对应分层 md 文件中的引用
   → 在 INDEX.md 导航区块中确认链接仍有效
```

### 迭代补充的质量约束

迭代补充的知识条目与初始建库条目遵守相同的质量标准，额外要求：

- 必须标注触发场景类型，便于后续分析知识库的薄弱环节
- 若补充内容与已有条目存在重叠，必须先做合并判断，不得重复建库
- 若补充内容与已有条目存在冲突，必须显式标注冲突并请用户决策，不得自行覆盖
- 代码推导来源的迭代补充条目，置信度默认为"中"，需用户确认后才能正式写入

### 知识库健康度感知

在每次完成较大规模 coding 任务后（如完成一个需求、修复一个复杂 bug），AI 可主动输出一次简短的知识库健康度提示：

```
📚 知识库健康度提示：
  本次任务中触发了 N 次无锚点推断，涉及 [层级] 的 [知识点]
  建议补充：[候选条目标题列表]
  如需现在补充，回复「补充知识库」；如需稍后处理，回复「跳过」
```
