# 单模块代码评审 (Module Code Review)

专注于单个模块的代码评审，确保代码质量符合项目规范和业务需求。

---

## ⚠️ 核心原则

1. **四文档对照**：必须对照项目知识库、技术方案、需求PRD、UI规格进行评审
2. **违规必须修复**：凡是违反项目知识库、技术方案、PRD、UI规格的，都必须修复
3. **聚焦当前模块**：仅评审当前开发的模块，不涉及其他模块
4. **问题分级**：问题分为「必须修复」「建议修复」「可选优化」三级
5. **给出修复方案**：每个问题必须提供具体的修复建议

---

## 输入信息

调用此 skill 时，需要提供以下信息：

| 参数 | 说明 | 必填 |
|------|------|------|
| task_id | 任务ID（如 T01） | 是 |
| task_title | 任务标题 | 是 |
| task_type | 任务类型（api/tool/hook/service/page/component） | 是 |
| module_path | 模块路径 | 是 |
| requirement_dir | 需求目录（如 .catpaw/docs/free-order-member-buy/） | 是 |
| techdoc_path | 技术方案路径 | 是 |
| prd_path | PRD文档路径 | 否（如存在则提供） |
| ui_spec_path | UI规格文件路径 | 否（UI任务必填） |

## 评审流程

### Step 1：收集评审依据

#### 1.1 读取项目知识库

**查找知识库目录**：

```bash
# 获取项目根目录
project_root=$(pwd)

# 查找项目根目录下所有 *-knowledge 目录
find "$project_root" -type d -name "*-knowledge" -maxdepth 1
```

**读取知识库索引**：

```bash
# 读取 INDEX.md
read_file {knowledge_dir}/INDEX.md
```

**根据任务类型读取相关知识**：

| 任务类型 | 必读知识文件 |
|---------|-------------|
| api | L0-spec.md, L2-modules.md, L6-experience.md |
| tool | L0-spec.md, L6-experience.md |
| hook | L0-spec.md, L1-architecture.md, L6-experience.md |
| service | L0-spec.md, L2-modules.md, L6-experience.md |
| page | L0-spec.md, L1-architecture.md, L2-modules.md, L6-experience.md |
| component | L0-spec.md, L1-architecture.md, L6-experience.md |

#### 1.2 读取技术方案

```bash
read_file {techdoc_path}
```

提取关键信息：
- 模块设计要求
- 接口契约
- 数据结构定义
- 技术实现方案

#### 1.3 读取需求PRD（如存在）

```bash
read_file {prd_path}
```

提取关键信息：
- 业务需求点
- 业务规则
- 边界条件
- 异常处理要求

#### 1.4 读取UI规格文件（UI任务）

```bash
read_file {ui_spec_path}
```

提取关键信息：
- 组件结构
- 样式规范
- 布局要求

---

### Step 2：读取被评审代码

#### 2.1 读取模块代码

```bash
read_file {module_path}
```

#### 2.2 获取代码差异（如适用）

```bash
# 获取当前分支与 master 的差异
git diff master -- {module_path}
```

---

### Step 3：执行评审检查

#### 3.1 项目知识库符合度检查

**检查项**：

| 检查项 | 检查内容 | 知识来源 |
|--------|----------|----------|
| 编码规范 | 命名规范、代码风格、文件组织 | L0-spec.md |
| 架构符合度 | 目录结构、模块划分、依赖关系 | L1-architecture.md |
| 业务逻辑 | 业务规则实现、边界处理 | L2-modules.md |
| 踩坑点规避 | 是否避开了已知问题 | L6-experience.md |

**检查方式**：
- 逐条对照 L0-spec.md 中的编码规范
- 验证代码结构是否符合 L1-architecture.md
- 检查业务逻辑实现是否符合 L2-modules.md 的描述
- 核对是否避开了 L6-experience.md 中记录的踩坑点

#### 3.2 技术方案一致性检查

**检查项**：

| 检查项 | 检查内容 |
|--------|----------|
| 接口契约 | API路径、方法、参数、返回值是否符合技术方案定义 |
| 数据结构 | 类型定义、字段名称、数据格式是否一致 |
| 实现方案 | 是否按照技术方案设计的方案实现 |
| 错误处理 | 异常处理是否符合技术方案要求 |

#### 3.3 PRD一致性检查（如有PRD）

**检查项**：

| 检查项 | 检查内容 |
|--------|----------|
| 业务需求 | 是否实现了PRD中定义的所有功能点 |
| 业务规则 | 条件判断、计算逻辑是否符合业务规则 |
| 边界条件 | 是否处理了所有边界场景 |
| 异常处理 | 错误提示、异常流程是否符合需求 |

#### 3.4 UI规格符合度检查（UI任务）

**检查项**：

| 检查项 | 检查内容 |
|--------|----------|
| 组件结构 | 是否按照UI规格定义的组件结构实现 |
| 样式还原 | 样式是否与设计稿一致 |
| 交互实现 | 交互行为是否符合设计要求 |

#### 3.5 代码质量检查

**检查项**：

| 检查项 | 检查内容 |
|--------|----------|
| 可读性 | 代码是否清晰易懂，命名是否准确 |
| 可维护性 | 是否有重复代码，是否易于修改 |
| 可测试性 | 是否易于编写单元测试 |
| 性能 | 是否存在性能问题（N+1查询、重复渲染等） |
| 安全性 | 是否存在安全隐患（XSS、注入等） |

## 评审结果处理

| 评审结果 | 处理方式 |
|---------|----------|
| 全部通过 | ✅ 返回「通过」，进入下一步骤 |
| 有违反知识库/技术方案/PRD/UI的问题 | ❌ 返回「需修改」，列出问题清单，必须修复 |
| 仅有「建议/可选」问题 | ⚠️ 返回「通过但有建议」，列出建议清单 |

**违规强制修复规则**：

| 违规来源 | 强制修复 | 说明 |
|---------|----------|------|
| 项目知识库（L0-L6） | ✅ 必须修复 | 违反项目规范、架构、业务逻辑、踩坑点规避 |
| 技术方案 | ✅ 必须修复 | 违反接口契约、数据结构、实现方案定义 |
| PRD需求文档 | ✅ 必须修复 | 业务需求遗漏、业务规则错误、边界条件缺失 |
| UI规格文件 | ✅ 必须修复 | 组件结构不符、样式还原度不足、交互实现错误 |
| 代码质量 | 🟡 建议修复 | 可读性、可维护性、性能等非规范性问题 |

## 注意事项

1. **违规零容忍**：凡是违反项目知识库、技术方案、PRD、UI规格的问题，必须修复，不接受「建议修复」或「可选优化」
2. **知识库优先**：评审依据以项目知识库为准，知识库未定义的规范不做强制要求
3. **需求溯源**：每个问题必须指出违反的具体规范或需求来源
4. **可操作性**：修复建议必须具体可执行，不能只说「需要优化」
5. **适度原则**：代码质量类问题（可读性、性能等）可作为建议，不影响功能正确性
