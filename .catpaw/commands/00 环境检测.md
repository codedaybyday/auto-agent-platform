# 项目知识库环境检测

检测当前项目的知识库环境，按以下决策树自动处理：

1. 检测 `@wmfe/spec-workflow` 全局包版本，非最新时自动更新
2. 检测 `@wmfe/ai-coding-stats` 全局包版本，非最新时更新，并执行 ONES 绑定
3. 若 `.catpaw/knowledge_hub/` 目录已存在 → 询问是否更新本地知识库（支持学城文档或 K-Hub ID）
4. 若目录不存在 → 调用 `layered-knowledge-builder` skill 生成新的本地知识库

### 1.2 检查并执行 ONES 绑定

在确保 `@wmfe/ai-coding-stats` 已安装最新版本后，执行 ONES 绑定流程：

#### Step 1.2.1: 询问用户是否需要绑定 ONES 任务

使用 AskQuestion 工具询问用户：

```json
{
  "questions": [
    {
      "id": "bind_ones",
      "prompt": "是否需要绑定 ONES 任务以便统计代码变更？",
      "options": [
        {
          "id": "yes",
          "label": "是，绑定 ONES 任务"
        },
        {
          "id": "no",
          "label": "否，跳过绑定"
        }
      ]
    }
  ]
}
```

#### Step 1.2.2: 如果用户选择绑定，询问 ONES 任务链接

使用 AskQuestion 工具询问用户：

```json
{
  "questions": [
    {
      "id": "ones_task_url",
      "prompt": "请提供 ONES 任务链接：",
      "options": [
        {
          "id": "paste",
          "label": "我已准备好粘贴链接"
        }
      ]
    }
  ]
}
```

等待用户粘贴 ONES 任务链接（格式如：https://ones.sankuai.com/ones/product/{space_id}/workItem/task/detail/{task_id}）

#### Step 1.2.3: 执行 ONES 绑定流程

用户提供任务链接后，按照 `spec-coding-stats` skill 中的 ONES 绑定流程执行：

**Step 1: 解析任务链接**

从链接中提取 `space_id` 和 `task_id`。

**Step 2: 获取任务信息**

优先使用 `web_fetch` 获取任务详情：

```
https://ones.sankuai.com/ones/product/{space_id}/workItem/task/detail/{task_id}
```

从返回数据中提取：
- `task_id`: `data.id.value`
- `task_name`: `data.name.value`
- `ones_url`: 任务链接本身
- `parentId.value`: 父需求 ID
- `developer`: `data.createdBy.value`（创建者 MIS ID）

**Step 3: 构造需求链接并获取需求信息**

使用 `parentId` 构造需求链接：

```
https://ones.sankuai.com/ones/product/{space_id}/workItem/requirement/detail/{requirement_id}
```

从返回数据中提取：
- `requirement_id`: `data.id.value`
- `requirement_name`: `data.name.value`
- `requirement_ones_url`: 构造的需求链接
- `start_time`: 使用 `ai-coding-stats formatTime` 转换 `data.createdAt.value`
- `end_time`: 使用 `ai-coding-stats formatTime` 转换 `data.expectClose.value`
- `prd_url`: 从 `data.description.displayValue` 中提取

**Step 4: 执行绑定命令**

```bash
ai-coding-stats bindOnes \
  --task_id "<task_id>" \
  --task_name "<task_name>" \
  --ones_url "<ones_url>" \
  --requirement_id "<requirement_id>" \
  --requirement_name "<requirement_name>" \
  --prd_url "<prd_url>" \
  --requirement_ones_url "<requirement_ones_url>" \
  --developer "<developer>" \
  --start_time "<start_time>" \
  --end_time "<end_time>"
```

输出绑定结果：

```text
## ONES 任务绑定完成

✅ 任务ID: {task_id}
✅ 任务名称: {task_name}
✅ 关联需求: {requirement_name}
✅ 绑定成功
```

**如果用户选择"否，跳过绑定"**：

```text
## ONES 绑定

状态: ⏭️ 已跳过

如需后续绑定，可使用 `ai-coding-stats bindOnes` 命令手动绑定。
```

## Step 2: 检查项目知识库目录

### 2.0 清理历史知识库目录（memory-bank-local）

> `memory-bank-local` 是旧版知识库目录名，已被 `knowledge_hub` 替代。如果仓库中存在该目录，需要删除以避免混淆。

```bash
# 检查是否存在旧版 memory-bank-local 目录
if [ -d ".catpaw/memory-bank-local" ]; then
  echo "FOUND_MEMORY_BANK:.catpaw/memory-bank-local"
elif [ -d "memory-bank-local" ]; then
  echo "FOUND_MEMORY_BANK:memory-bank-local"
else
  # 也搜索一下项目根目录下其他位置（排除 node_modules）
  FOUND_OLD=$(find . -maxdepth 2 -type d -name "memory-bank-local" -not -path "*/node_modules/*" 2>/dev/null | head -1)
  if [ -n "$FOUND_OLD" ]; then
    echo "FOUND_MEMORY_BANK:$FOUND_OLD"
  else
    echo "NO_MEMORY_BANK"
  fi
fi
```

**如果输出 `FOUND_MEMORY_BANK:<路径>`**：存在旧版知识库目录，直接删除：

```bash
# 删除旧版 memory-bank-local 目录
rm -rf {找到的路径}
```

输出清理结果：

```text
## 历史知识库清理

**检测到旧版目录**: {路径}
**操作**: 已删除（已迁移至 knowledge_hub）
✅ 历史知识库目录已清理
```

**如果输出 `NO_MEMORY_BANK`**：不存在旧版目录，跳过，继续下一步。

### 2.1 检查知识库目录是否存在

依次检查以下位置是否存在知识库目录（支持 `knowledge_hub` 或 `knowledge_hub` 命名）：

```bash
# 1. 首先检查标准位置
if [ -d ".catpaw/knowledge_hub" ]; then
  echo "FOUND:.catpaw/knowledge_hub"
else
  # 2. 搜索项目根目录下所有包含 knowledge 的目录（排除 node_modules 和 .catpaw/skills）
  FOUND_DIR=$(find . -maxdepth 2 -type d \( -name "knowledge_hub" -o -name "knowledge_hub" \) -not -path "*/node_modules/*" -not -path "*/.catpaw/skills/*" 2>/dev/null | head -1)
  if [ -n "$FOUND_DIR" ]; then
    echo "FOUND_OTHER:$FOUND_DIR"
  else
    echo "NOT_FOUND"
  fi
fi
```

**如果输出 `FOUND:.catpaw/knowledge_hub`**：知识库已在标准位置，进入 Step 3

**如果输出 `FOUND_OTHER:<路径>`**：知识库在其他位置，进入 Step 2.2 迁移流程

**如果输出 `NOT_FOUND`**：未找到知识库，进入 Step 4

### 2.2 知识库在其他位置 - 自动迁移到 .catpaw/knowledge_hub

检测到知识库位于非标准位置 `{找到的路径}`，自动迁移到标准位置 `.catpaw/knowledge_hub/`。

```bash

# 迁移知识库目录
```

迁移完成后进入 Step 3。

## 注意事项

1. **spec-workflow 版本检测**：
   - 检测全局安装的 `@wmfe/spec-workflow` 包版本
   - 非最新版本时提示用户更新
   - 更新后自动执行 `spec update` 拉取最新内容

2. **ai-coding-stats 版本检测**：
   - 检测全局安装的 `@wmfe/ai-coding-stats` 包版本
   - 未安装或版本过旧时自动安装或更新
   - 确保已安装最新版本后执行 ONES 绑定流程

3. **ONES 任务绑定**：
   - 询问用户是否需要绑定 ONES 任务
   - 如需绑定，收集任务链接并自动获取任务和需求信息
   - 执行 `ai-coding-stats bindOnes` 完成绑定

4. **知识库目录位置**：标准位置为 `.catpaw/knowledge_hub/`（位于项目根目录的 .catpaw 目录下）。如果检测到知识库在其他位置（如根目录的 `knowledge_hub/`），会提示迁移到标准位置

5. **生成位置**：知识库生成在 `.catpaw/knowledge_hub/` 文件夹

6. **知识库搜索范围**：搜索项目根目录下所有 `knowledge_hub` 或 `knowledge_hub` 目录（排除 `node_modules` 和 `.catpaw/skills`），支持自动发现非标准位置的知识库并迁移

7. **知识库更新来源**：
   - 学城文档：通过 KM 链接获取知识内容
   - K-Hub ID：从 K-Hub 平台拉取已有知识库

8. **skill 调用**：使用 `layered-knowledge-builder` skill 进行知识库生成、更新和同步

9. **K-Hub 平台**：K-Hub 是 BlockAI 的知识管理平台，用于存储和检索项目知识库

10. **完整性**：生成的知识库包含 11 个层级文件，覆盖项目的各个方面

## 错误处理

**package.json 不存在**：
- 提示用户切换到正确的项目根目录
- 确保当前目录包含 package.json 文件

**skill 调用失败**：
- 检查 .catpaw/skills/layered-knowledge-builder/ 目录是否存在
- 查看错误日志，根据提示修复问题

**知识库生成不完整**：
- 检查项目代码结构是否符合规范
- 确保有足够的代码文件供分析
