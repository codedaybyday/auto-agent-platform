# 技术方案全流程（techdoc-pipeline）

从需求文档出发，依次执行「生成技术方案 → 质量评审 → 问题修复」三个阶段，全自动产出经过评审和修复的高质量技术设计文档。

## 使用方式

```
/techdoc-pipeline <prd-file-path> [km-url]
```

示例：
- `/techdoc-pipeline .catpaw/docs/merchant-promo-perm/prd.md https://km.sankuai.com/collabpage/2747985261`
- `/techdoc-pipeline .catpaw/docs/merchant-promo-perm/prd.md`

## 参数说明

- `<prd-file-path>`（必填）：需求解析结果文件路径，通常位于 `.catpaw/docs/{需求英文名}/prd.md`
- `[km-url]`（可选）：原始 KM 文档链接，提供后可补充 prd.md 中未覆盖的需求细节

## 执行阶段

---

### 阶段一：生成技术方案

执行 `/techdoc-generator` skill，严格按照 skill 中的完整流程执行。

阶段一完成后，在对话中输出：
```
✅ 阶段一完成：技术方案已生成
📄 文件路径：{techdoc.md 完整路径}
➡️ 进入阶段二：质量评审...
```

---

### 阶段二：质量评审

**不等待用户确认，直接进入评审。**

执行 `/techdoc-reviewer` skill，严格按照 skill 中的完整流程执行（**跳过询问修复环节**）。

评审所需材料：
- 技术方案：阶段一生成的 `techdoc.md`
- PRD 文档：`<prd-file-path>`
- 原始 KM 文档（若提供了 `[km-url]`）

将评审报告写入与 `techdoc.md` 同目录的 `techdoc-review.md`。

阶段二完成后，在对话中输出：
```
✅ 阶段二完成：评审报告已生成
📄 文件路径：{techdoc-review.md 完整路径}
🔴 严重问题：X 个
🟡 一般问题：Y 个
➡️ 进入阶段三：自动修复...
```

---

### 阶段三：问题修复

**不等待用户确认，直接按评审报告修复技术方案。**

读取 `techdoc-review.md` 中的问题列表，根据评审报告中的改进建议，对 `techdoc.md` 进行修复：

**可自动修复的问题（直接修改）**：
- 🔴 接口字段缺失、类型不匹配、必填性错误、路径错误、HTTP 方法错误
- 🟡 命名不一致、枚举值遗漏、错误码未处理

**需人工确认的问题（标注后跳过）**：
- 功能点遗漏、需求理解偏差、技术栈选型冲突等涉及核心设计决策的问题
- 在 `techdoc.md` 对应位置添加 `<!-- ⚠️ 待人工确认：{问题描述} -->` 标注

**修复原则**：
- 每处修复添加标注：`<!-- 自动修复：{修复原因} -->`
- 只修复发现的问题，不主动优化其他内容
- 无法确定修复方式的，保留原样并标注「待确认」

修复完成后，同步更新 `techdoc-review.md` 中各问题的修复状态。

---

### 流程完成

三个阶段全部完成后，输出汇总：

```
🎉 技术方案全流程完成

📁 输出文件：
  - techdoc.md（已修复）：{完整路径}
  - techdoc-review.md（已更新）：{完整路径}

📊 修复汇总：
  - ✅ 自动修复：X 个问题
  - ⚠️ 待人工确认：Y 个问题（已在文档中标注）

💡 建议后续操作：
  - 检查文档中的 ⚠️ 标注，手动处理需人工确认的问题
  - 确认无误后可提交技术方案评审
```

---

## 错误处理

- 若 `<prd-file-path>` 文件不存在：告知用户文件路径有误，停止执行
- 若 skill 执行失败：告知用户具体错误原因，停止执行
- 若 `km` 命令不存在：先执行 `source ~/.meituan-local-tools/.venv/bin/activate`，若仍不可用则跳过原始文档读取，继续后续流程
- 若阶段一生成失败：停止流程，告知用户错误原因
- 若阶段二评审失败：保留阶段一产物，告知用户评审失败原因，停止流程
- 若阶段三修复过程中出错：保留已修复内容，告知用户哪些问题未能修复
