# Nine Raptor 监控埋点上报

为当前项目自动分析业务场景，结合技术方案文档中的稳定性保障方案（监控告警部分），生成并添加 Raptor 监控埋点上报代码。支持高达（gd）和 H5 两种技术栈。

## 前置检查

### 1. 确认 Nine MCP 已接入

检查当前环境是否已配置 Nine MCP（`@nine/nine-ai`）。如果未配置，提示用户参考 https://km.sankuai.com/collabpage/2707879881 进行接入。

### 2. 读取项目信息

读取当前工作目录下的 `package.json`，获取：
- `name` 字段作为 `componentName`
- 判断技术栈类型（高达项目通常包含 `@gundam/*` 系列依赖；H5 项目包含 `@nine/nine-preset-wm-h5`）

如果用户通过参数指定了技术栈（`gd` 或 `h5`），优先使用用户指定的值。

### 3. 确认目标目录

- 如果用户指定了目录名，使用该目录
- 否则默认扫描当前项目根目录下所有文件

---

## 执行流程

### Step 1: 收集项目信息

**获取当前分支和项目信息**：

```bash
git branch --show-current
git remote -v
```

**从项目知识库获取信息**：

检查 `knowledge_hub/` 目录是否存在，如果存在则读取索引文件了解知识库结构：

```bash
ls -la knowledge_hub/ 2>/dev/null && echo "===KNOWLEDGE_HUB_EXISTS===" || echo "===KNOWLEDGE_HUB_NOT_FOUND==="
```

如果知识库存在，读取 INDEX.md 了解知识库结构和检索决策树：

```bash
cat knowledge_hub/INDEX.md 2>/dev/null | head -100
```

**根据知识库结构读取相关文档**：

- L0-spec.md: 项目规范与约定
- L1-architecture.md: 工程架构
- L2-modules.md: 功能模块知识
- ORG-1.md / ORG-2.md: 组织级基建库

**如果知识库不存在或不完整，收集代码文件信息**：

```bash
find ./src -type f \( -name "*.ts" -o -name "*.tsx" -o -name "*.js" -o -name "*.jsx" -o -name "*.vue" \) 2>/dev/null | head -50
```

---

### Step 2: 读取技术方案文档

**1. 查找技术方案文档**：

```bash
find .catpaw/docs -name "techdoc.md" 2>/dev/null | head -5
```

**2. 提取稳定性保障方案内容**：

重点关注以下章节：
- **2.3.6 监控告警**：提取页面、技术栈、指标类型、指标项、告警项
- **2.3.5 容灾容错方案**：提取核心关键路径、依赖项、容灾方案
- **2.1.3 业务埋点**：提取已有的业务埋点规划

**3. 如果技术方案不存在**：

提示用户：
> 未找到技术方案文档，将仅基于项目代码结构生成埋点规划。建议先执行 `/techdoc-generator` 生成技术方案。

---

### Step 3: 检查埋点规范文件

**检查知识库中的埋点规范**：

优先检查以下位置：

1. **knowledge_hub/L0-spec.md** 中的埋点规范章节
2. **knowledge_hub/L2-modules.md** 中相关模块的埋点要求
3. **项目根目录下的 reportInfo/TrackingSpec.md**（如果存在）

```bash
# 检查知识库中的埋点规范
grep -l "埋点\|tracking\|raptor" knowledge_hub/*.md 2>/dev/null || echo "NO_TRACKING_SPEC_IN_KB"

# 检查传统埋点规范文件
ls -la reportInfo/TrackingSpec.md 2>/dev/null || echo "NO_TRADITIONAL_SPEC"
```

---

### Step 4: 执行埋点

#### 4.1 如果存在埋点规范文件

**立即执行埋点**：

```
nine track，帮我给当前这个【trackingType】项目【目录名（如有）】添加 raptor 监控上报代码。

【埋点规范】
依据知识库中的埋点规范执行：
- knowledge_hub/L0-spec.md 中的埋点规范章节
- knowledge_hub/L2-modules.md 中相关模块的埋点要求

【技术方案监控告警要求】
{从 Step 2 提取的监控告警表格内容}

【容灾容错监控需求】
{从 Step 2 提取的容灾容错相关监控需求}
```

参数映射：
- `trackingType`: 根据前置检查判断的技术栈，填入 `gd`（高达）或 `h5`
- `componentName`: 从 package.json 读取的项目名
- `codeContent`: Step 1 收集的文件列表
- `executeImmediately`: `true`（立即执行）
- `projectTrackingSpec`: `knowledge_hub/L0-spec.md`（或知识库中包含埋点规范的文件路径）

#### 4.2 如果不存在埋点规范文件

**直接执行埋点**（基于技术方案和项目知识）：

```
nine track，帮我给当前这个【trackingType】项目【目录名（如有）】添加 raptor 监控上报代码。

【项目知识库信息】
{从 Step 1 收集的项目知识库内容}

【技术方案监控告警要求】
{从 Step 2 提取的监控告警表格内容}

【容灾容错监控需求】
{从 Step 2 提取的容灾容错相关监控需求}
```

参数映射：
- `trackingType`: 根据前置检查判断的技术栈
- `componentName`: 从 package.json 读取的项目名
- `codeContent`: Step 1 收集的文件列表
- `executeImmediately`: `true`（立即执行）

---

### Step 5: 验证埋点代码

埋点代码生成后，执行以下验证：

1. **检查生成的埋点代码是否符合规范**：
   - type 和 name 字段使用英文字母、下划线、数字（不含中文）
   - type 数量控制在 200 个以内
   - 不使用可变的 id 类字段作为 type/name

2. **验证技术方案监控告警需求的覆盖情况**：
   - 对照技术方案中的「监控告警」表格，逐项确认是否已生成对应埋点
   - 对照「容灾容错方案」，确认降级/兜底场景是否有对应监控埋点
   - 标注未覆盖的监控项，提示需要补充

3. **检查埋点代码是否正确引入了 raptor 相关依赖**

4. **运行 lint 检查**（如有配置）：
   ```bash
   npm run lint 2>/dev/null || yarn lint 2>/dev/null || echo "no lint script found"
   ```

---

### Step 6: 输出结果报告

**确定需求目录**：

```bash
find .catpaw/docs -name "tasks.md" -o -name "techdoc.md" 2>/dev/null | xargs ls -t 2>/dev/null | head -1 | xargs dirname 2>/dev/null
```

**输出报告到文件**：`.catpaw/docs/{需求英文名}/track.md`

报告内容格式：

```markdown
# Nine Raptor 埋点上报 - 执行结果

> 执行时间: [YYYY-MM-DD HH:mm:ss]

**状态**: SUCCESS / PARTIAL / FAILED

**执行路径**: 直接执行 / 完整流程（生成规范后执行）

**项目信息**:
- 项目名: [componentName]
- 技术栈: [gd/h5]
- 扫描目录: [目录]
- 埋点规范: knowledge_hub/L0-spec.md（或知识库中包含埋点规范的文件路径）
- 技术方案: [techdoc.md 路径 或 "未关联"]

**埋点添加情况**:
- 新增埋点文件: [文件列表]
- 埋点类型数量: [count]
- 主要埋点场景: [场景描述]

**技术方案监控告警覆盖情况**（如有技术方案）:

| 监控类型 | 技术方案要求 | 埋点实现 | 覆盖状态 |
|---------|-------------|---------|----------|
| 性能监控 | [指标项] | [raptor type] | ✅已覆盖 / ⚠️需补充 |
| 稳定性监控 | [指标项] | [raptor type] | ✅已覆盖 / ⚠️需补充 |
| 业务监控 | [指标项] | [raptor type] | ✅已覆盖 / ⚠️需补充 |
| 接口监控 | [指标项] | [raptor type] | ✅已覆盖 / ⚠️需补充 |
| 容灾降级 | [场景] | [raptor type] | ✅已覆盖 / ⚠️需补充 |

**注意事项**:
- 请务必人工校验生成的埋点代码是否符合业务预期
- 确认 type/name 字段不含中文或特殊字符
- 建议在 Raptor 平台验证埋点数据上报是否正常
```

**同时在对话中输出简要结果**：

```
✅ 埋点上报完成！

📄 详细报告已保存至: .catpaw/docs/{需求英文名}/track.md
📊 埋点类型数量: [count]
📁 新增埋点文件: [文件数量] 个
📋 技术方案监控覆盖: [已覆盖项]/[总项数]（如有技术方案）

{如有未覆盖项，列出}
⚠️ 未覆盖的监控项:
- [监控项1]
- [监控项2]
```

### Step 7: 清理规划文档

**执行埋点完成后，删除生成的规划文档目录**：

```bash
# 删除 reportInfo 目录
rm -rf reportInfo
```

## 参数说明

| 参数 | 说明 | 示例 |
|------|------|------|
| 技术栈 | `gd`（高达/PC）或 `h5`（H5移动端） | `gd` |
| 目录名 | 可选，指定要添加埋点的子目录 | `src/pages/order` |

## 使用示例

```
/nine-raptor-track gd
/nine-raptor-track h5 src/pages/member
/nine-raptor-track
```

## 注意事项

- Nine MCP 需要提前配置，参考：https://km.sankuai.com/collabpage/2707879881
- 生成的埋点代码**必须人工校验**，确认符合业务预期后再提交
- 埋点 type/name 不支持中文，不支持特殊字符（ASCII 32-126 范围外）
- 单个应用 type 数量限制 400 个，每个 type 下 name 数量限制 400 个
- 如需添加 Ocean 业务埋点，请使用 `/nine-ocean-track` 命令
