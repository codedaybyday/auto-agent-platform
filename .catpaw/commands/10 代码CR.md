# 代码 CR（code-review）

基于 PRD 需求文档和技术方案，对当前分支相对于 master 的代码变更进行全面评审，检查业务逻辑实现、需求对齐度、技术规范符合性，生成结构化评审报告，并询问是否自动修复发现的问题。

## 使用方式

```
/code-review [prd-id-or-url] [techdoc-id-or-url] [branch]
```

示例：
- `/code-review`（自动从 .catpaw/docs 目录查找 prd.md 和 techdoc.md）
- `/code-review https://km.sankuai.com/page/123456 https://km.sankuai.com/page/789012`
- `/code-review 123456 789012`（直接使用学城页面 ID）
- `/code-review 123456 789012 feature/my-branch`（指定评审分支，默认为当前分支）

## 参数说明

- `[prd-id-or-url]`（可选）：PRD 需求文档，支持学城完整链接或页面 ID。不提供时自动从 `.catpaw/docs/*/prd.md` 查找
- `[techdoc-id-or-url]`（可选）：技术方案文档，支持学城完整链接或页面 ID。不提供时自动从 `.catpaw/docs/*/techdoc.md` 查找
- `[branch]`（可选）：指定要评审的分支，不提供时自动使用当前 Git 分支

## 执行步骤

### 第一步：参数解析与环境检查

解析命令参数：

1. **查找 PRD 文档**：
   - 若用户提供了 `prd-id-or-url` 参数：
     - 若为完整 URL（包含 `km.sankuai.com`）→ 直接使用
     - 若为纯数字 → 拼接为 `https://km.sankuai.com/page/{id}`
   - 若未提供参数，自动从项目目录查找：
     ```bash
     # 查找 .catpaw/docs 目录下的 prd.md 文件
     find .catpaw/docs -name "prd.md" -type f 2>/dev/null
     ```
     - 若找到多个文件，列出供用户选择
     - 若找到单个文件，直接使用该文件路径
     - 若未找到，提示用户需要提供 PRD 参数

2. **查找技术方案文档**：
   - 若用户提供了 `techdoc-id-or-url` 参数：
     - 若为完整 URL（包含 `km.sankuai.com`）→ 直接使用
     - 若为纯数字 → 拼接为 `https://km.sankuai.com/page/{id}`
   - 若未提供参数，自动从项目目录查找：
     ```bash
     # 查找 .catpaw/docs 目录下的 techdoc.md 文件
     find .catpaw/docs -name "techdoc.md" -type f 2>/dev/null
     ```
     - 若找到多个文件，列出供用户选择
     - 若找到单个文件，直接使用该文件路径
     - 若未找到，提示用户需要提供技术方案参数

3. 确认评审分支：
   ```bash
   git rev-parse --abbrev-ref HEAD
   ```
   - 若用户提供了 `[branch]` 参数 → 使用指定分支
   - 否则 → 使用当前分支输出结果

4. 检查 Git 状态：
   ```bash
   git status --short
   git log --oneline -3
   ```
   输出当前分支名和最近 3 条提交，让用户确认评审范围正确。

**若 PRD 和技术方案文档均未找到，立即停止并提示：**

```
⚠️ 缺少必要文档，无法开始 CR。

请提供 PRD 和技术方案文档，方式一：

  用法：/code-review [prd-id-or-url] [techdoc-id-or-url] [branch]

  示例：
    /code-review https://km.sankuai.com/page/123456 https://km.sankuai.com/page/789012
    /code-review 123456 789012

方式二：在 .catpaw/docs/{需求名}/ 目录下放置 prd.md 和 techdoc.md 文件

已搜索以下路径：
  - .catpaw/docs/*/prd.md
  - .catpaw/docs/*/techdoc.md
```

### 第二步：执行 code-review-analyzer skill

执行 `/code-review-analyzer` skill，严格按照 skill 中的完整流程执行。

将以下信息作为 skill 的输入：
- **PRD 文档**：第一步解析出的 PRD 链接
- **技术方案文档**：第一步解析出的技术方案链接
- **评审分支**：第一步确认的分支名

Skill 将自动完成：
- 获取学城 PRD 和技术方案文档内容
- 自动检测项目技术栈（nine / 高达 / React / Vue 等）
- 计算当前分支与 master 的代码 diff
- 进行多维度评审（需求对齐、业务逻辑、技术规范、代码质量、安全性、性能）
- 生成结构化评审报告，保存到 `.catpaw/docs/{需求名}/` 目录

**确定输出目录**：

根据 PRD 或技术方案文件的路径，确定需求名称：
```bash
# 假设 prd.md 或 techdoc.md 位于 .catpaw/docs/fake-dish-min-order/
# 则需求名为 fake-dish-min-order
```

评审报告输出位置：
```
.catpaw/docs/{需求名}/
├── prd.md           ← PRD 文档（已存在）
├── techdoc.md       ← 技术方案（已存在）
└── cr.md            ← 评审报告（本次生成）
```

### 第三步：展示评审摘要

Skill 执行完成后，在对话中输出评审摘要：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 代码 CR 完成
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
分支：{branch} → master
技术栈：{检测到的技术栈}
变更文件：{N} 个

评审评分：
  代码质量：{X} / 10
  业务逻辑实现：{X} / 10
  技术实现：{X} / 10

需求覆盖：
  ✅ 已实现：{N} 项（{X}%）
  ⚠️ 部分实现：{N} 项
  ❌ 未实现：{N} 项

问题汇总：
  🔴 必须修改：{N} 个
  🟡 建议修改：{N} 个
  🔵 可选优化：{N} 个

报告已保存至：{.catpaw/docs/{需求名}/cr.md 完整路径}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 第四步：询问是否自动修复

展示摘要后，询问用户：

```
是否需要自动修复评审中发现的问题？

  1. 全部修复 — 修复所有「必须修改」和「建议修改」的问题
  2. 仅修复严重问题 — 只修复「必须修改」（🔴）的问题
  3. 跳过 — 不自动修复，手动处理

请选择（1/2/3）：
```

根据用户选择执行：

**选择 1（全部修复）**：
- 按评审报告中「十、评审意见」的顺序，逐条修复「必须修改」和「建议修改」的问题
- 每修复一个问题，在对话中输出：`✅ 已修复：{问题标题}（{文件路径}:{行号}）`
- 所有修复完成后，重新运行测试（若项目有测试命令）：
  ```bash
  # 检测测试命令
  cat package.json | grep -E '"test"|"jest"|"vitest"' | head -3
  ```
  若存在测试命令，运行 `npm test` 或 `npm run test`，确认修复未引入新问题
- 输出修复总结：已修复 {N} 个问题，跳过 {N} 个（需人工处理）

**选择 2（仅修复严重问题）**：
- 只修复评审报告「10.1 必须修改」章节中的问题
- 每修复一个问题，在对话中输出：`✅ 已修复：{问题标题}（{文件路径}:{行号}）`
- 修复完成后输出：已修复 {N} 个严重问题，{N} 个建议修改问题需手动处理

**选择 3（跳过）**：
- 不执行任何修复
- 输出提示：「评审报告已保存，可随时查阅 {报告路径} 手动处理问题。」

### 修复注意事项

执行自动修复时：

- **每次只修复一个问题**，修复后立即验证（运行相关测试或 lint）
- **不修改测试文件**，除非评审报告明确指出测试逻辑有误
- **遇到无法自动修复的问题**（如需要与后端联调确认的接口问题、需要产品确认的业务逻辑问题），跳过并在总结中注明「需人工处理」
- **修复前备份**：若修改超过 5 个文件，先执行 `git stash` 备份当前状态，再开始修复

### 错误处理

| 错误情况 | 处理方式 |
|---------|---------|
| 学城文档无法访问 | 提示用户检查链接或权限，停止执行 |
| 当前分支无 diff（与 master 相同） | 提示「当前分支与 master 无差异，无需 CR」，停止执行 |
| Git 不在仓库中 | 提示「当前目录不是 Git 仓库」，停止执行 |
| `km` 命令不存在 | 先执行 `source ~/.meituan-local-tools/.venv/bin/activate`，若仍不可用则提示用户手动提供文档内容 |
| 修复过程中测试失败 | 立即停止修复，回滚该文件的修改（`git checkout -- {file}`），在总结中标注「修复失败，需人工处理」 |

### 成功标准

- 评审报告 `cr.md` 已写入 `.catpaw/docs/{需求名}/` 目录
- 在对话中输出了评审摘要（评分、问题数量、需求覆盖率）
- 用户已选择是否自动修复，并按选择执行完毕
- 告知用户报告的完整路径
